## 1. Spring Framework 核心源码

### 1.1 IoC 容器

#### BeanFactory 与 ApplicationContext 继承体系

```
BeanFactory（最底层接口）
    └── HierarchicalBeanFactory（支持层级）
        └── AutowireCapableBeanFactory（支持自动装配）
            └── AbstractAutowireCapableBeanFactory（抽象实现）
                └── DefaultListableBeanFactory（默认实现，核心）

ApplicationContext（扩展接口）
    └── ConfigurableApplicationContext（可配置）
        └── AbstractApplicationContext（抽象实现）
            └── ClassPathXmlApplicationContext
            └── AnnotationConfigApplicationContext
            └── SpringApplication（Spring Boot 入口）
```

**BeanFactory**：提供基本的 IoC 和 DI 功能，延迟加载（懒加载）。
**ApplicationContext**：继承 BeanFactory，扩展了事件传播、国际化、资源加载、AOP 集成等功能，默认预实例化单例 Bean。

#### BeanFactoryPostProcessor 与 BeanPostProcessor 的作用时机

```
容器启动时序：
┌─────────────────────────────────────────────────────────┐
│  1. 加载 BeanDefinition（XML/注解扫描）                    │
│  2. 执行 BeanFactoryPostProcessor.postProcessBeanFactory() │
│     → 可修改 BeanDefinition（如 PropertyPlaceholderConfigurer）│
│  3. 注册 BeanPostProcessor                                   │
│  4. 实例化单例 Bean                                          │
│     4.1 实例化（createBeanInstance）                        │
│     4.2 填充属性（populateBean）                             │
│     4.3 初始化（initializeBean）                             │
│         → BeanPostProcessor.postProcessBeforeInitialization() │
│         → 执行 init-method / @PostConstruct                  │
│         → BeanPostProcessor.postProcessAfterInitialization()  │
│  5. 容器就绪                                                 │
└─────────────────────────────────────────────────────────┘
```

**关键区别**：
- `BeanFactoryPostProcessor`：在 Bean 实例化之前执行，操作 BeanDefinition
- `BeanPostProcessor`：在 Bean 实例化之后执行，操作 Bean 实例（AOP 代理在此创建）

#### 单例 Bean 的创建流程（doCreateBean）

`AbstractApplicationContext.refresh()` 是容器启动的"上帝方法"，它会调用 `finishBeanFactoryInitialization(beanFactory)`，最终走到 `DefaultListableBeanFactory.preInstantiateSingletons()`。这里开始对每一个非懒加载的单例 Bean 调用 `getBean(beanName)`。

**AbstractBeanFactory.doGetBean() 逻辑**：
1. **尝试从缓存获取**：先查单例池（`singletonObjects`），如果查到直接返回
2. **检查循环依赖**：如果缓存没有，去当前正在创建的 Bean 集合中查找（`singletonsCurrentlyInCreation`）
3. **处理依赖**：如果存在依赖，可能会暴露早期引用（三级缓存）
4. **调用 createBean**：进入真正的创建流程

**doCreateBean 的五个核心步骤**：

```java
protected Object doCreateBean(final String beanName, final RootBeanDefinition mbd, 
                              final @Nullable Object[] args) {
    // 1. 实例化：通过反射或工厂方法创建对象，此时属性全为null
    BeanWrapper instanceWrapper = createBeanInstance(beanName, mbd, args);
    Object bean = instanceWrapper.getWrappedInstance();

    // 2. 允许提前暴露：将 ObjectFactory（生成早期引用）放入三级缓存
    //    这是解决循环依赖的关键
    addSingletonFactory(beanName, () -> getEarlyBeanReference(beanName, mbd, bean));

    // 3. 填充属性：处理 @Autowired、@Value 等依赖注入
    populateBean(beanName, mbd, instanceWrapper);

    // 4. 初始化：执行 Aware、BeanPostProcessor 的 before/after、init-method
    exposedObject = initializeBean(beanName, exposedObject, mbd);

    // 5. 注册可销毁 Bean 的回调
    registerDisposableBeanIfNecessary(beanName, bean, mbd);

    return exposedObject;
}
```

**关键点**：第二步的"提前暴露"并不马上执行 `getEarlyBeanReference`，而是将一个能生成早期引用的工厂放入 `singletonFactories`，当有别的 Bean 需要依赖它时，会触发这个工厂来获取早期引用（可能已被 AOP 代理包裹）。

#### 循环依赖解决：三级缓存

**什么是三级缓存？**

Spring 的单例缓存有三个 Map：

| 缓存级别 | 名称 | 内容 | 状态 |
|---------|------|------|------|
| 一级 | `singletonObjects` | 完全初始化好的单例 Bean | 成品 |
| 二级 | `earlySingletonObjects` | 已实例化但尚未填充属性的早期 Bean | 半成品 |
| 三级 | `singletonFactories` | 用于生成早期引用的 ObjectFactory | 工厂 |

**为什么需要三级缓存？**

假设 A 依赖 B，B 依赖 A 的场景：

1. 创建 A，实例化后将其 ObjectFactory 放入三级缓存
2. 填充 A 的属性时发现需要 B，于是去创建 B
3. 创建 B 时，B 也需要 A，此时从三级缓存中取出 A 的 ObjectFactory，调用 `getObject()` 生成 A 的早期引用（可能是 CGLIB/JDK 代理）。这个早期引用会被提升到二级缓存，并从三级缓存中删除
4. B 拿到 A 的早期引用后，完成自己的属性填充和初始化，最终成为完全体，放入一级缓存
5. 回到 A 的创建流程，A 此时可以直接从一级缓存获取完全的 B，然后完成自身初始化，最后将 A 的最终版放入一级缓存

**两级缓存行不行？**

如果只有二级缓存（存放早期对象），当 A 需要被 AOP 代理时，普通 Bean 的早期对象与最终的代理对象不同，会导致注入的早期引用与最终暴露给外界的 Bean 不一致。三级缓存中的 `ObjectFactory` 可以在生产早期引用的瞬间，通过 `getEarlyBeanReference` 方法生成代理对象，保证即使提前暴露也是同一个代理实例。

**@Autowired 的提前暴露机制**：

`@Autowired` 注入时，Spring 会调用 `doResolveDependency`，最终委托给 `DefaultListableBeanFactory.getBean`。因此它天然走上述缓存流程。循环依赖能够解决的前提是：**Bean 必须是单例，且不能是构造器注入**。因为构造器注入在实例化时就需要传入依赖，此时连实例都还没生成，无法放入缓存。

**@Lazy 解决构造器循环依赖**：

```java
@Component
public class A {
    private B b;

    @Autowired
    public A(@Lazy B b) {  // @Lazy 注入代理对象
        this.b = b;
    }
}
```

`@Lazy` 让依赖注入变为一个代理对象，真正使用时才去容器获取，避免了创建时的死循环。

#### 💡 面试点睛

**面试题 1：Spring 如何解决循环依赖？**

回答要点：提及三级缓存（`singletonObjects` / `earlySingletonObjects` / `singletonFactories`），描绘 A/B 互相依赖的流程，说明 `ObjectFactory` 的作用，以及为何不能是构造器注入。

**面试题 2：二级缓存能解决循环依赖吗？为什么要三级？**

回答：如果没有 AOP，二级其实够用；但有 AOP 时，早期引用必须是代理对象，三级缓存中的 `getEarlyBeanReference` 会统一生成代理，保证提前暴露的引用和最终代理一致。如果去掉三级，只能在创建 Bean 一开始就生成代理（侵入性强且违背 Bean 生命周期）。

**面试题 3：@Lazy 注解能解决构造器循环依赖吗？**

回答：能，因为它让依赖注入变为一个代理对象，真正使用时才去容器获取，避免了创建时的死循环。

**面试题 4：Spring Bean 的生命周期是怎样的？**

回答：实例化（createBeanInstance）→ 属性填充（populateBean）→ 初始化（initializeBean：Aware 接口 → BeanPostProcessor.before → init-method → BeanPostProcessor.after）→ 使用 → 销毁（DisposableBean.destroy / @PreDestroy）。

**面试题 5：BeanFactoryPostProcessor 和 BeanPostProcessor 的区别？**

回答：BeanFactoryPostProcessor 在 Bean 实例化之前执行，操作 BeanDefinition（如修改配置属性）；BeanPostProcessor 在 Bean 实例化之后执行，操作 Bean 实例（如 AOP 代理创建）。

### 1.2 AOP 原理

#### @EnableAspectJAutoProxy → AnnotationAwareAspectJAutoProxyCreator

```java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Import(AspectJAutoProxyRegistrar.class)
public @interface EnableAspectJAutoProxy {
    boolean proxyTargetClass() default false;  // 是否强制 CGLIB
}
```

`AspectJAutoProxyRegistrar` 实现了 `ImportBeanDefinitionRegistrar`，向容器注册 `AnnotationAwareAspectJAutoProxyCreator`（一个 `BeanPostProcessor`）。

#### 代理创建判断

```java
// AbstractAutoProxyCreator.postProcessAfterInitialization()
@Override
public Object postProcessAfterInitialization(@Nullable Object bean, String beanName) {
    if (bean != null) {
        // 1. 获取缓存 Key
        Object cacheKey = getCacheKey(bean.getClass(), beanName);
        // 2. 检查是否已经处理过
        if (this.earlyProxyReferences.remove(cacheKey) != bean) {
            // 3. 判断是否需要代理：findEligibleAdvisors
            return wrapIfNecessary(bean, beanName, cacheKey);
        }
    }
    return bean;
}
```

**findEligibleAdvisors → getAdvicesAndAdvisorsForBean**：
1. 从容器中获取所有 Advisor（包括 @Aspect 切面中的 @Before、@After 等）
2. 通过 `Pointcut` 匹配，筛选出适用于当前 Bean 的 Advisor
3. 如果有匹配的 Advisor，则创建代理

#### JDK 动态代理 vs CGLIB 的选择规则

| 条件 | 代理方式 |
|------|---------|
| 目标类实现了接口 | JDK 动态代理（默认） |
| 目标类无接口 | CGLIB |
| `proxyTargetClass = true` | 强制 CGLIB |
| 目标类是 final | 无法代理（CGLIB 不能继承 final 类） |

**JDK 动态代理**：
```java
// 核心类：JdkDynamicAopProxy
// 实现 InvocationHandler，代理所有接口方法
public Object invoke(Object proxy, Method method, Object[] args) {
    // 1. 获取方法对应的拦截器链
    List<Object> chain = this.advised.getInterceptorsAndDynamicInterceptionAdvice(...);
    // 2. 创建 ReflectiveMethodInvocation
    MethodInvocation invocation = new ReflectiveMethodInvocation(proxy, target, method, args, targetClass, chain);
    // 3. 执行调用链
    return invocation.proceed();
}
```

**CGLIB 代理**：
```java
// 核心类：CglibAopProxy
// 生成目标类的子类，重写非 final 方法
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(rootClass);
enhancer.setCallback(new DynamicAdvisedInterceptor(advised));
```

#### MethodInterceptor 调用链

`ReflectiveMethodInvocation.proceed()` 是 AOP 拦截链的核心：

```java
public Object proceed() throws Throwable {
    // 如果拦截器执行完毕，执行目标方法
    if (this.currentInterceptorIndex == this.interceptorsAndDynamicMethodMatchers.size() - 1) {
        return invokeJoinpoint();
    }

    // 获取下一个拦截器
    Object interceptorOrInterceptionAdvice = 
        this.interceptorsAndDynamicMethodMatchers.get(++this.currentInterceptorIndex);

    // 执行拦截器（如 MethodBeforeAdviceInterceptor、AspectJAfterAdvice 等）
    return ((MethodInterceptor) interceptorOrInterceptionAdvice).invoke(this);
}
```

**拦截器类型**：
- `MethodBeforeAdviceInterceptor`：@Before
- `AspectJAfterAdvice`：@After
- `AspectJAfterThrowingAdvice`：@AfterThrowing
- `AspectJAroundAdvice`：@Around

#### 💡 面试点睛

**面试题 1：Spring AOP 的实现原理是什么？**

回答：Spring AOP 基于动态代理实现。目标类有接口时使用 JDK 动态代理（实现 InvocationHandler），无接口时使用 CGLIB（生成子类）。通过 BeanPostProcessor 在 Bean 初始化后创建代理对象，代理对象内部维护拦截器链（MethodInterceptor），通过 ReflectiveMethodInvocation.proceed() 递归执行。

**面试题 2：JDK 动态代理和 CGLIB 的区别？**

回答：JDK 动态代理基于接口，通过反射调用目标方法，只能代理接口方法；CGLIB 基于继承，通过 ASM 生成子类字节码，可以代理类的方法（但不能代理 final 方法）。JDK 代理创建速度快，CGLIB 执行速度略快。

**面试题 3：@Transactional 失效的场景有哪些？**

回答：（1）非 public 方法；（2）同类内部方法调用（this 调用不走代理）；（3）异常被 catch 未抛出；（4）rollbackFor 配置不当（默认只回滚 RuntimeException）；（5）数据库引擎不支持事务（如 MyISAM）；（6）多数据源未指定事务管理器。

**面试题 4：同一个类中方法 A 调用方法 B，B 上的 @Transactional 会生效吗？**

回答：不会。因为 this.B() 是目标对象直接调用，不经过代理对象，所以事务拦截器不会生效。解决方案：注入自身代理对象调用，或使用 AspectJ 编译期织入。

### 1.3 事务管理

#### @Transactional 的解析与代理增强

```java
@Transactional(propagation = Propagation.REQUIRED, isolation = Isolation.DEFAULT)
public void transfer(String from, String to, BigDecimal amount) {
    // 业务逻辑
}
```

**解析流程**：
1. `SpringTransactionAnnotationParser` 解析 `@Transactional` 注解
2. `TransactionAttributeSource` 存储事务属性（传播行为、隔离级别、超时、只读等）
3. `TransactionInterceptor` 作为方法拦截器，在目标方法执行前后管理事务

#### TransactionInterceptor 与 PlatformTransactionManager

```java
public class TransactionInterceptor extends TransactionAspectSupport implements MethodInterceptor {
    @Override
    public Object invoke(MethodInvocation invocation) throws Throwable {
        Class<?> targetClass = (invocation.getThis() != null ? AopUtils.getTargetClass(invocation.getThis()) : null);
        return invokeWithinTransaction(invocation.getMethod(), targetClass, invocation::proceed);
    }
}
```

**事务执行流程**：
```
1. 获取事务属性（TransactionAttribute）
2. 获取 PlatformTransactionManager（如 DataSourceTransactionManager）
3. 判断是否存在事务：
   - 无事务 → 根据传播行为创建新事务（begin）
   - 有事务 → 根据传播行为处理（加入/挂起/嵌套）
4. 执行目标方法
5. 方法成功 → commit
6. 方法异常 → rollback（根据 rollbackFor 判断）
```

#### 事务传播行为的实现原理

| 传播行为 | 含义 | 实现 |
|---------|------|------|
| `REQUIRED`（默认） | 当前无事务则新建，有则加入 | `handleExistingTransaction` 或 `startTransaction` |
| `REQUIRES_NEW` | 挂起当前事务，新建独立事务 | `suspend` 当前事务，`startTransaction` 新事务 |
| `NESTED` | 在当前事务中创建嵌套事务（保存点） | `createSavepoint` |
| `SUPPORTS` | 有事务则加入，无则不使用 | 检查 `TransactionSynchronizationManager` |
| `NOT_SUPPORTED` | 挂起当前事务，以非事务执行 | `suspend` + 无事务执行 |
| `MANDATORY` | 必须在事务中执行，否则抛异常 | 检查 `TransactionSynchronizationManager` |
| `NEVER` | 必须在非事务中执行，否则抛异常 | 检查 `TransactionSynchronizationManager` |

#### 嵌套事务与保存点（Savepoint）

```java
// NESTED 传播行为使用保存点
Connection conn = dataSource.getConnection();
Savepoint savepoint = conn.setSavepoint("nested-sp");
try {
    // 执行嵌套事务逻辑
    conn.commit();
} catch (Exception e) {
    conn.rollback(savepoint);  // 只回滚到保存点，不影响外层事务
}
```

**注意**：`NESTED` 与 `REQUIRES_NEW` 的区别：
- `NESTED`：嵌套事务回滚不影响外层事务，但外层事务回滚会连带回滚嵌套事务
- `REQUIRES_NEW`：完全独立的事务，互不影响

#### 💡 面试点睛

**面试题 1：Spring 事务的传播行为有哪些？默认是什么？**

回答：7 种传播行为，默认 REQUIRED。常用的是 REQUIRED（加入/新建）、REQUIRES_NEW（挂起当前新建独立事务）、NESTED（嵌套保存点）。

**面试题 2：REQUIRED 和 REQUIRES_NEW 的区别？**

回答：REQUIRED 会加入当前事务，与外层事务共进退；REQUIRES_NEW 会挂起当前事务，开启全新独立事务，外层事务回滚不影响它，它回滚也不影响外层。

**面试题 3：事务的隔离级别有哪些？Spring 默认是什么？**

回答：DEFAULT（跟随数据库）、READ_UNCOMMITTED、READ_COMMITTED、REPEATABLE_READ、SERIALIZABLE。Spring 默认 DEFAULT，MySQL 默认 REPEATABLE_READ，Oracle 默认 READ_COMMITTED。

**面试题 4：Spring 事务实现原理？**

回答：通过 AOP 代理实现。解析 @Transactional 生成 TransactionInterceptor，在方法执行前开启事务（begin），方法成功提交（commit），方法异常回滚（rollback）。底层使用 ThreadLocal 绑定 Connection，保证同一线程内多个 DAO 操作共享同一连接。

### 1.4 Spring MVC 请求流程

#### DispatcherServlet 核心方法 doDispatch

```java
protected void doDispatch(HttpServletRequest request, HttpServletResponse response) {
    HttpServletRequest processedRequest = request;
    HandlerExecutionChain mappedHandler = null;

    try {
        ModelAndView mv = null;
        Exception dispatchException = null;

        // 1. 检查文件上传
        processedRequest = checkMultipart(request);

        // 2. 获取 HandlerExecutionChain（Handler + Interceptors）
        mappedHandler = getHandler(processedRequest);
        if (mappedHandler == null) {
            noHandlerFound(processedRequest, response);
            return;
        }

        // 3. 获取 HandlerAdapter
        HandlerAdapter ha = getHandlerAdapter(mappedHandler.getHandler());

        // 4. 执行拦截器 preHandle
        if (!mappedHandler.applyPreHandle(processedRequest, response)) {
            return;
        }

        // 5. 执行 Handler（Controller 方法）
        mv = ha.handle(processedRequest, response, mappedHandler.getHandler());

        // 6. 应用默认视图名
        applyDefaultViewName(processedRequest, mv);

        // 7. 执行拦截器 postHandle
        mappedHandler.applyPostHandle(processedRequest, response, mv);

        // 8. 处理结果（渲染视图或异常）
        processDispatchResult(processedRequest, response, mappedHandler, mv, dispatchException);
    } catch (Exception ex) {
        // 触发拦截器 afterCompletion
        triggerAfterCompletion(processedRequest, response, mappedHandler, ex);
    }
}
```

#### 完整请求处理链

```
HTTP Request
    ↓
DispatcherServlet.doDispatch()
    ↓
1. HandlerMapping（RequestMappingHandlerMapping）
   → 根据 @RequestMapping 解析 URL，找到对应的 HandlerMethod
    ↓
2. HandlerAdapter（RequestMappingHandlerAdapter）
   → 适配 HandlerMethod，准备参数解析、返回值处理
    ↓
3. 参数解析（HandlerMethodArgumentResolver）
   → @RequestParam、@PathVariable、@RequestBody、@ModelAttribute 等
    ↓
4. 执行 Controller 方法
    ↓
5. 返回值处理（HandlerMethodReturnValueHandler）
   → @ResponseBody → MessageConverter（Jackson）→ JSON
   → String → ViewResolver → 视图渲染
    ↓
6. 异常处理（@ExceptionHandler / HandlerExceptionResolver）
    ↓
HTTP Response
```

#### @RequestMapping 的解析与匹配

**解析时机**：容器启动时，`RequestMappingHandlerMapping` 扫描所有 `@Controller` 类，解析 `@RequestMapping` 注解，构建 `RequestMappingInfo` 对象。

**匹配过程**：
```java
// RequestMappingInfo.getMatchingCondition()
// 按以下顺序匹配：
1. Path → @RequestMapping("/users/{id}")
2. Method → GET/POST/PUT/DELETE
3. Params → params="name=zhang"
4. Headers → headers="Content-Type=application/json"
5. Consumes → consumes="application/json"
6. Produces → produces="application/json"
```

**面试题 1：Spring MVC 的请求处理流程？**

回答：DispatcherServlet.doDispatch() → getHandler() 通过 HandlerMapping 找到 Handler → getHandlerAdapter() 获取适配器 → applyPreHandle() 执行拦截器 → handle() 执行 Controller 方法 → applyPostHandle() → processDispatchResult() 处理视图或异常 → triggerAfterCompletion()。

**面试题 2：@RequestMapping 和 @GetMapping 的区别？**

回答：@GetMapping 是 @RequestMapping(method = RequestMethod.GET) 的缩写，是组合注解。Spring 4.3 引入，代码更简洁。

**面试题 3：Spring MVC 的拦截器和 Filter 的区别？**

回答：（1）拦截器是 Spring 的，Filter 是 Servlet 的；（2）拦截器可以访问 Spring 容器中的 Bean，Filter 不行；（3）Filter 在请求进入 Servlet 之前执行，拦截器在 Handler 执行前后执行；（4）Filter 基于函数回调，拦截器基于反射（AOP）。

**面试题 4：@Controller 和 @RestController 的区别？**

回答：@RestController = @Controller + @ResponseBody，所有方法默认返回 JSON/XML 等数据，不走视图解析器。@Controller 方法返回 String 时会解析为视图名。

    // 1. 先检查本地缓存
    Class<?> clazz = findLoadedClass0(name);
    if (clazz != null) return clazz;

    // 2. 检查系统类（避免 Web 应用覆盖核心类）

    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
        // 1. 检查是否有足够字节（长度字段）
        if (in.readableBytes() < 4) return;

        // 2. 读取长度
        int length = in.readInt();

            return;

    }
}

### 4.4 内存管理

#### ByteBuf 的引用计数与池化

```java
// 创建 ByteBuf（池化、堆外内存）
ByteBuf buf = PooledByteBufAllocator.DEFAULT.directBuffer(1024);

// 引用计数 +1
buf.retain();

// 使用...

// 引用计数 -1，为 0 时释放
buf.release();
```

**引用计数优势**：
- 精确控制内存生命周期
- 避免 GC 带来的停顿
- 支持跨 Handler 传递而不复制

#### PooledByteBufAllocator 的内存分配机制

```
PooledByteBufAllocator
    ├── PoolArena（Direct / Heap）
    │   ├── PoolChunk（16MB，由 2048 个 8KB Page 组成）
    │   │   └── PoolSubpage（小对象分配：≤ 8KB）
    │   └── PoolChunkList（按使用率分类）
    │       ├── qInit（0-25%）
    │       ├── q000（1-50%）
    │       ├── q025（25-75%）
    │       ├── q050（50-100%）
    │       └── q075（75-100%）
```

**分配策略**：
- 小对象（≤ 8KB）：从 PoolSubpage 分配（类似 jemalloc 的 slab 分配）
- 大对象（> 8KB）：从 PoolChunk 分配连续 Page
- 巨型对象（> 16MB）：非池化分配

#### 堆外内存的释放与监控

```java
// 监控堆外内存使用
long directMemoryUsed = PooledByteBufAllocator.DEFAULT.metric().usedDirectMemory();

// 手动释放（确保引用计数归 0）
ReferenceCountUtil.release(buf);
```

**注意**：堆外内存不受 GC 管理，如果 `ByteBuf` 泄漏（未 `release`），会导致 `OutOfDirectMemoryError`。

**面试题 1：Netty 的线程模型是什么？**

回答：主从 Reactor 多线程模型。bossGroup（1 个线程）负责 accept 连接，workerGroup（默认 CPU 核数 * 2）负责 IO 读写和业务处理。一个 Channel 只绑定一个 EventLoop，保证单线程执行，无锁化。

**面试题 2：Netty 的 ByteBuf 相比 Java NIO 的 ByteBuffer 有什么优势？**

回答：（1）读写指针分离，不需要 flip()；（2）支持池化（PooledByteBufAllocator），减少 GC；（3）引用计数精确控制内存释放；（4）支持 CompositeByteBuf 组合零拷贝；（5）容量自动扩展。

**面试题 3：Netty 的内存池是怎么设计的？**

回答：借鉴 jemalloc 设计，分为 PoolArena（Direct/Heap）→ PoolChunk（16MB）→ PoolSubpage（小对象 ≤8KB）。小对象从 PoolSubpage 分配，大对象分配连续 Page，巨型对象非池化。使用 PoolChunkList 按使用率分类管理。

**面试题 4：Netty 的零拷贝有哪些实现方式？**

回答：（1）FileRegion：文件传输直接通过 FileChannel.transferTo() 从内核到网卡；（2）CompositeByteBuf：组合多个 ByteBuf 不复制数据，底层使用 writev 系统调用；（3）Slice/Duplicate：共享底层数组，只创建新的视图。

**面试题 5：Netty 的 ChannelPipeline 是怎么工作的？**

回答：ChannelPipeline 是双向链表，包含 inbound 和 outbound 两种 Handler。inbound 事件（如 channelRead）从 Head 向 Tail 传播，outbound 事件（如 write）从 Tail 向 Head 传播。通过 ctx.fireChannelRead() 和 ctx.writeAndFlush() 触发事件传递。

### 5.1 Dubbo 核心

#### 服务暴露与引用

**面试题 1：Dubbo 的 SPI 和 Java SPI 有什么区别？**

### 5.2 RocketMQ 核心

#### 消息存储结构

```
CommitLog（消息物理存储）
    ├── 顺序写，1GB 一个文件
    └── 存储消息体 + 元数据

ConsumeQueue（消费逻辑队列）
    ├── 每个 Topic 的每个 Queue 一个文件
    └── 存储 CommitLog Offset + Size + Tag HashCode

IndexFile（索引文件）
    ├── 按 Key 或时间范围查询
    └── Hash 索引 + 链表解决冲突
```

**为什么快？**
- CommitLog 顺序写，磁盘顺序写性能接近内存
- ConsumeQueue 定长条目（20 字节），可快速定位
- MMap 文件映射，读写通过内存操作

#### 事务消息实现原理

```
1. 生产者发送 Half Message（半消息，对消费者不可见）
2. 生产者执行本地事务
3. 生产者发送 Commit 或 Rollback
   - Commit：半消息对消费者可见
   - Rollback：删除半消息
4. Broker 定时回查（Producer 实现 checkLocalTransaction）
```

#### 💡 面试点睛

**面试题 1：RocketMQ 为什么快？**

回答：CommitLog 顺序写（磁盘顺序写性能接近内存），ConsumeQueue 定长条目（20 字节）快速定位，使用 MMap 内存映射文件，读写通过内存操作。所有 Topic 共享 CommitLog，避免随机写。

**面试题 2：RocketMQ 的事务消息是怎么实现的？**

回答：两阶段提交。第一阶段发送 Half Message（对消费者不可见），第二阶段根据本地事务结果发送 Commit 或 Rollback。Broker 定时回查未决消息，Producer 实现 checkLocalTransaction 接口返回事务状态。

#### 💡 面试点睛

**面试题 1：Jedis 和 Lettuce 的区别？**

回答：Jedis 是阻塞 IO，线程不安全需连接池；Lettuce 基于 Netty 非阻塞 IO，线程安全单连接多线程，原生支持异步和 Reactive API，性能更高。

**面试题 2：Redis Pipeline 和事务的区别？**

回答：Pipeline 是客户端批量发送命令、批量接收结果，不保证原子性；事务（MULTI/EXEC）是服务端保证原子性执行，但事务中的命令会排队执行，不利用 Pipeline 的并行优势。Redis 事务不支持回滚。

**面试题 3：Redis 的持久化方式有哪些？**

回答：RDB（快照，定时 fork 子进程 dump 内存到文件，恢复速度快但可能丢数据）和 AOF（追加日志，记录每个写命令，数据更安全但文件大、恢复慢）。Redis 4.0 支持混合持久化（RDB + AOF）。

### 5.4 ZooKeeper 分布式协调

#### 会话机制与 Watcher

```java
// 创建会话
ZooKeeper zk = new ZooKeeper("localhost:2181", 3000, watcher);

// Watcher 是一次性的，触发后需重新注册
zk.getData("/path", event -> {
    if (event.getType() == EventType.NodeDataChanged) {
        // 数据变化，重新获取并注册 Watcher
        zk.getData("/path", this, null);
    }
}, null);
```

#### ZAB 协议与 Leader 选举

**ZAB（ZooKeeper Atomic Broadcast）**：
1. **崩溃恢复**：Leader 选举（FastLeaderElection）
2. **消息广播**：Leader 接收写请求，生成 Proposal，Follower 投票，过半数提交

**Leader 选举**：
- 每个 Server 投票（myid, zxid）
- 比较规则：zxid 大的优先，zxid 相同则 myid 大的优先
- 收到过半票数则成为 Leader

#### Curator 分布式锁实现

```java
// Curator 可重入分布式锁
InterProcessMutex lock = new InterProcessMutex(client, "/locks/order");
if (lock.acquire(10, TimeUnit.SECONDS)) {
    try {
        // 执行业务逻辑
    } finally {
        lock.release();
    }
}

// 原理：
// 1. 在 /locks/order 下创建临时顺序节点（如 lock-00000001）
// 2. 检查是否是最小序号，是则获取锁
// 3. 否则监听前一个节点，等待其删除
// 4. 锁释放时删除节点，触发下一个节点的 Watcher
```

#### 💡 面试点睛

**面试题 1：ZooKeeper 的 Watcher 机制有什么特点？**

回答：Watcher 是一次性的，触发后需重新注册。这是为了减轻服务端压力。Watcher 只能保证最终一致性，不保证实时性（有网络延迟）。

**面试题 2：ZooKeeper 的 Leader 选举算法？**

回答：FastLeaderElection。每个 Server 投票（myid, zxid），比较规则：zxid 大的优先（数据越新），zxid 相同则 myid 大的优先。收到过半票数则成为 Leader。

**面试题 3：ZooKeeper 分布式锁和 Redis 分布式锁的区别？**

回答：ZooKeeper 通过临时顺序节点 + Watcher 实现，锁释放时自动删除节点（会话断开也删除），可靠性高但性能较低；Redis 通过 SETNX + 过期时间实现，性能高但需处理锁续期、主从同步延迟等问题。ZooKeeper 适合强一致性场景，Redis 适合高性能场景。

### 5.5 Spring Boot 自动装配原理

#### @SpringBootApplication 与 @EnableAutoConfiguration

```java
@SpringBootApplication  // 组合注解
├── @SpringBootConfiguration  // @Configuration
├── @EnableAutoConfiguration  // 启用自动装配
│   └── @Import(AutoConfigurationImportSelector.class)
└── @ComponentScan  // 组件扫描
```

#### AutoConfigurationImportSelector 与 spring.factories

```java
public class AutoConfigurationImportSelector implements DeferredImportSelector {
    @Override
    public String[] selectImports(AnnotationMetadata metadata) {
        // 1. 从 META-INF/spring.factories 加载自动配置类
        List<String> configurations = SpringFactoriesLoader.loadFactoryNames(
            EnableAutoConfiguration.class, getBeanClassLoader()
        );

        // 2. 过滤（@Conditional 条件判断）
        configurations = filter(configurations, autoConfigurationMetadata);

        // 3. 返回需要导入的配置类
        return StringUtils.toStringArray(configurations);
    }
}
```

**spring.factories 示例**：
```properties
# META-INF/spring.factories
org.springframework.boot.autoconfigure.EnableAutoConfiguration=org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration,org.springframework.boot.autoconfigure.web.servlet.DispatcherServletAutoConfiguration
```

#### 条件注解 @Conditional 系列源码分析

| 注解 | 条件 | 源码 |
|------|------|------|
| `@ConditionalOnClass` | 类路径存在指定类 | `OnClassCondition` |
| `@ConditionalOnMissingBean` | 容器中不存在指定 Bean | `OnBeanCondition` |
| `@ConditionalOnProperty` | 配置属性匹配 | `OnPropertyCondition` |
| `@ConditionalOnWebApplication` | 是 Web 应用 | `OnWebApplicationCondition` |

```java
// @ConditionalOnClass 实现
class OnClassCondition extends SpringBootCondition {
    @Override
    public ConditionOutcome getMatchOutcome(ConditionContext context, AnnotatedTypeMetadata metadata) {
        // 使用 ASM 读取注解元数据，不加载类
        MultiValueMap<String, Object> attributes = metadata.getAllAnnotationAttributes(
            ConditionalOnClass.class.getName()
        );
        // 检查类路径是否存在指定类
        return ClassNameFilter.MISSING.matches(names, getClass().getClassLoader()) 
            ? ConditionOutcome.noMatch("missing") 
            : ConditionOutcome.match();
    }
}
```

#### 💡 面试点睛

**面试题 1：Spring Boot 的自动装配原理？**

回答：@EnableAutoConfiguration 导入 AutoConfigurationImportSelector，从 META-INF/spring.factories 加载自动配置类列表，通过 @Conditional 条件过滤（如类路径是否存在、Bean 是否存在、配置属性是否匹配），将满足条件的配置类导入 Spring 容器。

**面试题 2：spring.factories 和 @Import 的区别？**

回答：spring.factories 是 SPI 机制，框架通过它发现扩展；@Import 是显式导入配置类。Spring Boot 2.7+ 推荐使用 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 替代 spring.factories。

**面试题 3：如何自定义 Starter？**

回答：（1）创建自动配置类，用 @Conditional 控制条件；（2）在 META-INF/spring.factories 中注册；（3）打包为独立 jar；（4）其他项目引入依赖即可自动装配。注意：配置类应提供 @EnableConfigurationProperties 绑定配置属性。
