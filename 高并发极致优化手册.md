## 1. 无锁编程与 CAS 深度

### 1.1 CAS 的 CPU 指令实现（cmpxchg）与 ABA 问题

#### 什么是 CAS

CAS（Compare-And-Swap）是一种原子操作，包含三个操作数：**内存位置 V**、**预期值 A**、**新值 B**。只有当 V 的值等于 A 时，才将 V 更新为 B，否则不做任何操作。

#### CPU 指令实现

在 x86 架构中，CAS 对应 `LOCK CMPXCHG` 指令：

```asm
lock cmpxchg [mem], new_value
```

- `LOCK` 前缀：锁定总线或缓存行，保证操作的原子性
- `CMPXCHG`：比较并交换
- 现代 CPU 对 `LOCK` 前缀做了优化，使用 **缓存一致性协议（MESI）** 而非总线锁定，性能大幅提升

#### ABA 问题

**场景**：线程 T1 读取值为 A，准备 CAS 时，线程 T2 将 A 改为 B 又改回 A，T1 的 CAS 仍然成功，但中间状态已改变。

```
时间线：
T1: 读取 V = A
T2: V = A → B（修改）
T2: V = B → A（改回）
T1: CAS(V, A, C) → 成功！但中间经历了 B
```

**影响**：链表/栈等数据结构操作中，可能导致节点已被释放但被误认为有效。

**解决方案**：

1. **AtomicStampedReference**：增加版本号
   ```java
   AtomicStampedReference<Integer> ref = new AtomicStampedReference<>(100, 0);
   int[] stampHolder = new int[1];
   Integer value = ref.get(stampHolder);
   ref.compareAndSet(value, 200, stampHolder[0], stampHolder[0] + 1);
   ```

2. **AtomicMarkableReference**：增加布尔标记

3. **业务层面规避**：避免复用对象（如使用唯一 ID）

#### CAS 的底层源码

```java
// Unsafe 中的 CAS 实现
public final native boolean compareAndSwapLong(Object o, long offset, long expected, long x);
```

在 HotSpot VM 中，`compareAndSwapLong` 对应 `unsafe.cpp` 中的 `Unsafe_CompareAndSwapLong`，最终调用 `Atomic::cmpxchg`。

#### 💡 面试点睛

**面试题 1：CAS 的底层 CPU 指令是什么？**

回答：x86 架构中是 `LOCK CMPXCHG` 指令。`LOCK` 前缀保证原子性，现代 CPU 使用缓存一致性协议（MESI）优化，而非总线锁定。`CMPXCHG` 比较内存值和预期值，相等则交换。

**面试题 2：什么是 ABA 问题？怎么解决？**

回答：ABA 问题是线程 T1 读取值 A，T2 将 A 改为 B 又改回 A，T1 的 CAS 仍然成功但中间状态已变。解决：使用 AtomicStampedReference（增加版本号）或 AtomicMarkableReference（增加布尔标记），或者业务层面避免对象复用。

**面试题 3：CAS 有什么缺点？**

回答：（1）ABA 问题；（2）高并发下自旋消耗 CPU（空转）；（3）只能保证一个变量的原子性，无法保证多个变量的原子操作（需用 synchronized 或 Lock）。

### 1.2 VarHandle（Java 9+）比 Unsafe 更安全的操作

#### 为什么需要 VarHandle

`sun.misc.Unsafe` 是 JDK 内部 API，不推荐使用，且可能在未来的 JDK 版本中移除。Java 9 引入 `VarHandle`，提供标准化的、安全的原子/内存操作。

#### VarHandle 核心能力

| 能力 | 方法 | 说明 |
|------|------|------|
| 普通读写 | `get/set` | 普通字段访问 |
| volatile 读写 | `getVolatile/setVolatile` | 保证可见性 |
| 有序读写 | `getOpaque/setOpaque` | 弱于 volatile，禁止指令重排 |
| 获取/释放 | `getAcquire/setRelease` | acquire/release 语义 |
| CAS | `compareAndSet` | 原子比较并交换 |
| 原子加减 | `getAndAdd` | 原子累加 |

#### VarHandle 使用示例

```java
import java.lang.invoke.MethodHandles;
import java.lang.invoke.VarHandle;

public class VarHandleDemo {
    private volatile int counter = 0;

    // 获取 VarHandle
    private static final VarHandle COUNTER;
    static {
        try {
            COUNTER = MethodHandles.lookup()
                .findVarHandle(VarHandleDemo.class, "counter", int.class);
        } catch (Exception e) {
            throw new Error(e);
        }
    }

    public void increment() {
        // 原子累加，等价于 AtomicInteger.getAndIncrement()
        COUNTER.getAndAdd(this, 1);
    }

    public int get() {
        return (int) COUNTER.getVolatile(this);
    }
}
```

#### VarHandle vs Unsafe

| 特性 | Unsafe | VarHandle |
|------|--------|-----------|
| 访问控制 | 需要反射绕过 | 标准 API，无需 hack |
| 类型安全 | 无（raw pointer） | 有（编译时类型检查）|
| 性能 | 极高 | 接近 Unsafe |
| 可移植性 | 差（内部 API） | 好（标准 API）|
| 功能范围 | 极广（内存分配、线程调度等）| 限于字段/数组操作 |

**建议**：新代码优先使用 VarHandle，遗留代码逐步迁移。

#### 💡 面试点睛

**面试题 1：VarHandle 和 Unsafe 有什么区别？**

回答：VarHandle 是 Java 9 的标准 API，类型安全、可移植，无需反射绕过；Unsafe 是内部 API，功能更广（内存分配、线程调度等），但可能在未来版本移除。新代码优先用 VarHandle，性能接近 Unsafe。

**面试题 2：VarHandle 的 getAcquire/setRelease 是什么语义？**

回答：getAcquire 保证读取操作不会被重排到后续的读/写操作之前；setRelease 保证写入操作不会被重排到之前的读/写操作之后。比 volatile 更轻量，适合实现无锁队列等场景。

- **入口断点**：打在 `java.lang.invoke.VarHandle` 的 `getAndAdd()` 方法，观察调用链
- **配合 JMH**：使用 JMH 基准测试对比 VarHandle 和 Unsafe 的性能差异
- **推荐资源**：JEP 193（Variable Handles）、OpenJDK 源码（VarHandle 实现）

### 1.3 无锁栈/队列的简单实现

#### 无锁栈（Treiber Stack）

```java
import java.util.concurrent.atomic.AtomicReference;

public class LockFreeStack<T> {
    private final AtomicReference<Node<T>> top = new AtomicReference<>();

    private static class Node<T> {
        final T value;
        Node<T> next;
        Node(T value) { this.value = value; }
    }

    public void push(T value) {
        Node<T> newNode = new Node<>(value);
        Node<T> oldTop;
        do {
            oldTop = top.get();
            newNode.next = oldTop;
        } while (!top.compareAndSet(oldTop, newNode));  // CAS 直到成功
    }

    public T pop() {
        Node<T> oldTop;
        Node<T> newTop;
        do {
            oldTop = top.get();
            if (oldTop == null) return null;
            newTop = oldTop.next;
        } while (!top.compareAndSet(oldTop, newTop));
        return oldTop.value;
    }
}
```

**注意**：此实现存在 ABA 问题（节点被 pop 后又 push 回来），生产环境建议使用 `AtomicStampedReference`。

#### 无锁队列（Michael-Scott Queue）

```java
import java.util.concurrent.atomic.AtomicReference;

public class LockFreeQueue<T> {
    private final Node<T> dummy = new Node<>(null);
    private final AtomicReference<Node<T>> head = new AtomicReference<>(dummy);
    private final AtomicReference<Node<T>> tail = new AtomicReference<>(dummy);

    private static class Node<T> {
        volatile T value;
        volatile AtomicReference<Node<T>> next;
        Node(T value) { this.value = value; this.next = new AtomicReference<>(); }
    }

    public void enqueue(T value) {
        Node<T> newNode = new Node<>(value);
        AtomicReference<Node<T>> curTail;
        AtomicReference<Node<T>> residue;

        while (true) {
            curTail = tail.get();
            residue = curTail.get().next;

            if (curTail == tail.get()) {
                if (residue.get() == null) {
                    if (curTail.get().next.compareAndSet(null, newNode)) {
                        tail.compareAndSet(curTail.get(), newNode);
                        return;
                    }
                } else {
                    tail.compareAndSet(curTail.get(), residue.get());
                }
            }
        }
    }
}
```

#### 💡 面试点睛

**面试题 1：如何实现一个无锁栈？**

回答：使用 AtomicReference 指向栈顶节点，push 和 pop 都用 CAS 循环直到成功。注意 ABA 问题，生产环境用 AtomicStampedReference。核心思想：CAS 保证原子性，失败则重试。

**面试题 2：无锁队列相比阻塞队列有什么优势？**

回答：无锁队列没有线程阻塞和唤醒的开销，没有锁竞争，适合极高并发场景。但实现复杂（需处理 ABA、多节点 CAS），且通常是无界队列。阻塞队列（如 LinkedBlockingQueue）实现简单，支持有界，适合生产者-消费者模式。

#### 源码阅读实战建议

- **入口断点**：打在 `java.util.concurrent.ConcurrentLinkedQueue` 的 `offer()` 方法，观察无锁队列实现
- **配合 jcstress**：编写并发测试验证无锁队列的正确性
- **推荐书籍**：《Java 并发编程实战》第 15 章（非阻塞算法）

### 1.4 CAS 自旋的性能代价与适用场景

#### 自旋的代价

CAS 失败后会循环重试（自旋），在高并发下：
- **CPU 空转**：消耗 CPU 周期但无实际进展
- **缓存一致性风暴**：多个 CPU 核心频繁修改同一缓存行，导致总线/缓存行 ping-pong
- **活锁**：线程不断 CAS 失败，但都不退让

#### 自旋优化策略

1. **指数退避**：失败后等待随机时间再重试
   ```java
   int backoff = 1;
   while (!cas(...)) {
       Thread.sleep(backoff);
       backoff = Math.min(backoff * 2, MAX_BACKOFF);
   }
   ```

2. **自适应自旋**：JVM 根据历史成功率动态调整自旋次数（`-XX:+UseSpinning`）

3. **分段分散**：如 LongAdder 将单一变量拆分为多个 Cell

#### 适用场景

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 低竞争（线程数 < CPU 核数）| CAS 自旋 | 开销小，响应快 |
| 高竞争（线程数 >> CPU 核数）| LongAdder / 锁 | 避免自旋浪费 |
| 需要精确值 | AtomicLong | LongAdder 的 sum() 不精确 |
| 读多写少 | StampedLock 乐观读 | 避免读锁开销 |

#### 💡 面试点睛

**面试题 1：CAS 自旋有什么性能问题？**

回答：高并发下 CAS 失败会循环重试，导致 CPU 空转；多个核心频繁修改同一缓存行，引发缓存一致性风暴（ping-pong）。优化：指数退避、自适应自旋、分段分散（如 LongAdder）。

**面试题 2：什么场景适合 CAS，什么场景不适合？**

回答：低竞争（线程数 < CPU 核数）适合 CAS，响应快；高竞争（线程数 >> CPU 核数）不适合，自旋浪费 CPU，应改用 LongAdder 或锁。需要精确值（如库存扣减）用 AtomicLong，统计指标用 LongAdder。

#### 源码阅读实战建议

- **入口断点**：打在 `java.util.concurrent.atomic.AtomicInteger` 的 `incrementAndGet()` 方法，观察 CAS 循环
- **配合 JMH**：使用 JMH 测试不同竞争程度下 CAS 和锁的性能差异
- **JVM 参数**：`-XX:+UseSpinning` 开启自适应自旋，观察性能变化

## 2. LongAdder 与 Striped64 原理

### 2.1 AtomicLong 的竞争瓶颈

在并发编程中，我们经常需要统计计数，比如 QPS、接口调用次数。你可能首先想到 `AtomicLong`，它通过 CAS 实现无锁更新，但是在高并发下 CAS 会频繁失败并自旋，导致 CPU 空转，性能急剧下降。

**简单对比测试（16线程，累加1亿次）**：

| 实现 | 耗时 |
|------|------|
| AtomicLong | 约 8,000 ms |
| LongAdder | 约 1,200 ms |

差异巨大。

#### 💡 面试点睛

**面试题 1：AtomicLong 在高并发下为什么性能差？**

回答：AtomicLong 基于单一变量的 CAS，高并发下多个线程竞争同一变量，CAS 频繁失败并自旋，导致 CPU 空转和缓存一致性风暴。性能随线程数增加而急剧下降。

---

### 2.2 LongAdder 的分段累加设计：base + Cell 数组

`LongAdder` 是 JDK 8 引入的原子累加器，它在多线程环境下比 `AtomicLong` 有更高的吞吐量。其核心思想是**分段（Striping）**：将热点数据拆分成多个 Cell，每个线程在自己对应的 Cell 上累加，最后汇总。这样线程间竞争分散，性能大幅提升。

#### 核心数据结构

```java
// 伪代码，省略部分细节
abstract class Striped64 extends Number {
    // 基础值，无竞争时直接 CAS 更新
    transient volatile long base;

    // Cell 数组，大小为 2 的幂，非竞争时为空
    transient volatile Cell[] cells;

    // 自旋锁标志，用于 cells 数组的初始化和扩容
    transient volatile int cellsBusy;

    // 内嵌的 Cell 类，使用 @Contended 防止伪共享
    @sun.misc.Contended static final class Cell {
        volatile long value;
        Cell(long x) { value = x; }

        final boolean cas(long cmp, long val) {
            return UNSAFE.compareAndSwapLong(this, valueOffset, cmp, val);
        }
    }
}
```

#### 💡 面试点睛

**面试题 1：LongAdder 为什么比 AtomicLong 快？**

回答：LongAdder 将单一变量拆分为 base + Cell[] 数组，线程通过 ThreadLocalRandom.probe 哈希映射到不同 Cell，分散竞争。无竞争时直接 CAS base，有竞争时各线程在各自 Cell 上累加，最后 sum() 汇总。避免了多线程竞争同一变量的 CAS 自旋。

### 2.3 源码解读：add()、sum()、reset()

#### add() 的过程

```java
public void add(long x) {
    Cell[] as; long b, v; int m; Cell a;
    // 如果 cells 不为空 或者 直接 CAS base 失败，进入 longAccumulate
    if ((as = cells) != null || !casBase(b = base, b + x)) {
        boolean uncontended = true;
        // 如果 cells 为空 或者 当前线程的探测值映射到的 Cell 不存在 或者 对 Cell CAS 失败
        if (as == null || (m = as.length - 1) < 0 ||
            (a = as[getProbe() & m]) == null ||
            !(uncontended = a.cas(v = a.value, v + x)))
            // 进入核心累加方法（Striped64 中）
            longAccumulate(x, null, uncontended);
    }
}
```

**流程解释**：
1. 如果 cells 没有初始化（无竞争历史），直接尝试 CAS 更新 base
2. 若 base CAS 失败，说明存在竞争，则进入 `longAccumulate` 处理

#### longAccumulate —— 竞争时的自适应路由

这是分段累加的精髓。它会为当前线程分配一个 Cell，或者对 Cell 进行扩容，或者在自旋中不断尝试。

**核心步骤**：
1. **初始化 cells 表**：如果 cells == null，尝试获取 cellsBusy 锁，然后创建大小为 2 的 Cell 数组
2. **分配 Cell**：通过 `ThreadLocalRandom.current()` 的探测值 probe 计算当前线程的 Cell 索引 `(probe & (n-1))`
3. **如果目标 Cell 为空**：尝试获取锁并新建一个 Cell 放入
4. **如果目标 Cell CAS 失败**：说明该 Cell 竞争也激烈，可能需要扩容（最大到 CPU 核数）
5. **扩容机制**：表大小翻倍，直到 >= NCPU。扩容时不影响其他 Cell

线程探测值 `probe` 是 `ThreadLocalRandom` 为每个线程维护的一个伪随机数，用于决定访问哪个 Cell，类似 HashMap 的 hash。每次线程竞争失败，probe 会重新生成（相当于 rehash），分散碰撞。

#### sum() 方法

```java
public long sum() {
    Cell[] as = cells; long sum = base;
    if (as != null) {
        for (Cell a : as)
            if (a != null)
                sum += a.value;
    }
    return sum;
}
```

**关键点**：`sum()` 并不是原子快照，计算时可能有其他线程并发更新，所以高并发下 `sum()` 的结果可能略偏小，不是精确值。LongAdder 适合统计指标，不能用于需要精确值的场景（如剩余库存）。

#### reset() 与 sumThenReset()

```java
public void reset() {
    Cell[] as = cells; long b = base;
    base = 0L;
    if (as != null) {
        for (Cell a : as)
            if (a != null)
                a.value = 0L;
    }
}

public long sumThenReset() {
    Cell[] as = cells; long sum = base;
    base = 0L;
    if (as != null) {
        for (Cell a : as) {
            if (a != null) {
                sum += a.value;
                a.value = 0L;
            }
        }
    }
    return sum;
}
```

`reset()` 会将所有 Cell 和 base 置零，`sumThenReset()` 是先获取 sum 再重置，同样不保证原子性。

#### 💡 面试点睛

**面试题 1：LongAdder 的 add() 方法流程是怎样的？**

回答：先尝试 CAS 更新 base，成功则返回；失败说明有竞争，获取当前线程的 probe 哈希映射到 Cell 数组的某个 Cell，对该 Cell CAS 累加。如果 Cell 为空或 CAS 失败，进入 longAccumulate 处理：初始化 Cell 数组、分配 Cell、扩容（最大到 CPU 核数）。

**面试题 2：LongAdder 的 sum() 为什么不精确？**

回答：sum() 遍历 base 和所有 Cell 时不加锁，其他线程可能正在更新。因此 sum() 不是原子快照，结果可能略偏小。LongAdder 适合统计场景（如 QPS），不适合需要精确值的业务（如库存扣减）。

**面试题 3：LongAdder 的扩容机制是怎样的？**

回答：当某个 Cell 竞争激烈（CAS 失败）时，会触发扩容。Cell 数组大小翻倍（2→4→8...），直到达到 CPU 核数。扩容时使用 cellsBusy 自旋锁保证线程安全，扩容后线程的 probe 重新哈希映射到新 Cell。

### 2.4 对比测速与使用注意（sum() 的高开销）

#### 性能测试

```java
import java.util.concurrent.*;
import java.util.concurrent.atomic.*;

public class LongAdderDemo {
    public static void main(String[] args) throws Exception {
        int threads = 16;
        long perThread = 100_000_000;

        // AtomicLong 测试
        AtomicLong atomic = new AtomicLong();
        ExecutorService pool = Executors.newFixedThreadPool(threads);
        long start = System.currentTimeMillis();
        for (int i = 0; i < threads; i++) {
            pool.execute(() -> {
                for (long j = 0; j < perThread; j++) {
                    atomic.incrementAndGet();
                }
            });
        }
        pool.shutdown();
        pool.awaitTermination(1, TimeUnit.HOURS);
        System.out.println("AtomicLong time: " + (System.currentTimeMillis() - start));

        // LongAdder 测试
        LongAdder adder = new LongAdder();
        ExecutorService pool2 = Executors.newFixedThreadPool(threads);
        start = System.currentTimeMillis();
        for (int i = 0; i < threads; i++) {
            pool2.execute(() -> {
                for (long j = 0; j < perThread; j++) {
                    adder.increment();
                }
            });
        }
        pool2.shutdown();
        pool2.awaitTermination(1, TimeUnit.HOURS);
        System.out.println("LongAdder time: " + (System.currentTimeMillis() - start));
    }
}
```

**输出示例（JDK 11，4核8线程）**：
```
AtomicLong time: 7245 ms
LongAdder time: 923 ms
```

性能相差近 8 倍。

#### 使用注意

1. **sum() 的弱一致性**：在并发更新时调用 sum() 可能返回一个比实际值小的数。不能用于需要精确计数的业务（如订单 ID 生成、库存扣减）。

2. **LongAdder 不是 AtomicLong 的通用替代**：LongAdder 没有 `compareAndSet` 等 CAS 操作，也不能拿到当前值的快照做逻辑判断，它纯粹是一个累加器。

3. **初始竞争高可能导致大量 Cell 创建**：如果一开始就有大量线程同时 `add()`，`longAccumulate` 会不断扩容，可能瞬间创建 CPU 核心数的 Cell，并且每个 Cell 用 `@Contended` 填充，内存占用较大。但稳定后性能极佳。

4. **错误的重置方式**：重置时如果有并发累加，可能导致部分累加丢失。如果必须重置，建议用 `sumThenReset()` 或在业务上容忍非精确。

5. **在高并发下大量求和**：频繁调用 `sum()` 会遍历 Cell 数组，如果线程很多且 Cell 数量多，会有一定开销。

#### 💡 面试点睛

**面试题 1：AtomicLong 和 LongAdder 有什么区别？什么场景用哪个？**

回答：AtomicLong 基于单一变量 CAS，适合低并发或需要 compareAndSet 场景（如库存扣减）。LongAdder 分段累加，吞吐量高，但 sum() 有弱一致性，适合统计指标（QPS、计数器）。如果业务需要精确的原子加减和条件更新，必须用 AtomicLong。

**面试题 2：LongAdder 的 sum() 为什么不是准确的？**

回答：sum() 遍历 base 和所有 Cell 时不加锁，其他线程可能正在更新。因此 LongAdder 主要用于统计场景，追求最终一致性和高吞吐。

**面试题 3：LongAdder 的内存占用大吗？**

回答：初始无竞争时只有 base（8 字节），竞争后创建 Cell 数组，每个 Cell 用 @Contended 填充（约 128 字节），最大到 CPU 核数。高并发下内存占用较大，但稳定后性能极佳。如果对内存敏感，需权衡。

## 3. 伪共享与 @Contended 注解

### 3.1 CPU 缓存行与伪共享现象

#### CPU 缓存架构

现代 CPU 采用多级缓存（L1/L2/L3），数据以**缓存行（Cache Line）**为单位传输，通常为 **64 字节**。当 CPU 读取一个变量时，会将整个缓存行加载到缓存中。

```
CPU Core 0          CPU Core 1          CPU Core 2          CPU Core 3
   |                    |                    |                    |
   L1 Cache            L1 Cache            L1 Cache            L1 Cache
   |                    |                    |                    |
   +--------------------+--------------------+--------------------+
                        |
                       L3 Cache (共享)
                        |
                      主内存
```

#### 伪共享（False Sharing）

**场景**：两个线程分别修改同一个缓存行中的不同变量。

```java
public class FalseSharingDemo {
    // volatile long 占 8 字节
    public volatile long value1 = 0L;
    // value2 与 value1 可能在同一个缓存行（64字节内）
    public volatile long value2 = 0L;
}
```

**问题**：
- Thread 1 修改 value1 → 使 Core 0 的缓存行失效
- Thread 2 修改 value2 → 使 Core 1 的缓存行失效
- 虽然两个线程修改不同变量，但因为共享缓存行，导致**缓存行在多个核心间来回失效（ping-pong）**
- 性能急剧下降（可能下降 10 倍以上）

#### 💡 面试点睛

**面试题 1：什么是伪共享（False Sharing）？**

回答：两个线程修改不同变量，但这两个变量在同一个缓存行（64 字节）中。一个线程修改后，另一个核心的缓存行失效，导致缓存行在多个核心间来回 ping-pong，性能急剧下降（可能 10 倍以上）。

**面试题 2：缓存行大小是多少？为什么重要？**

回答：现代 CPU 缓存行通常是 64 字节。数据以缓存行为单位传输，理解缓存行大小对优化内存布局、避免伪共享至关重要。JDK 中可以用 `@Contended` 或填充字段来避免伪共享。

---

### 3.2 填充技巧（p1, p2, p3...）

#### 手动填充

通过在变量前后填充无用字段，确保目标变量独占一个缓存行：

```java
public class PaddedLong {
    // 前填充：7 个 long = 56 字节
    public long p1, p2, p3, p4, p5, p6, p7;

    // 目标变量（8 字节）
    public volatile long value = 0L;

    // 后填充：7 个 long = 56 字节
    public long p9, p10, p11, p12, p13, p14, p15;
}
```

总大小：56 + 8 + 56 = 120 字节，确保 value 独占一个 64 字节缓存行（实际编译器可能优化掉未使用的字段，需配合 `@Contended`）。

#### JDK 7 的填充方式

```java
class Cell {
    volatile long p0, p1, p2, p3, p4, p5, p6;
    volatile long value;
    volatile long q0, q1, q2, q3, q4, q5, q6;
}
```

#### 💡 面试点睛

**面试题 1：手动填充伪共享有什么注意事项？**

回答：填充字段可能被编译器优化掉（认为未使用），需配合 `@Contended` 注解或确保字段被读取。填充会增加内存占用（每个变量额外约 128 字节），大量使用时需注意内存成本。

### 3.3 @sun.misc.Contended 原理与 JVM 参数

#### @Contended 注解

Java 8 引入 `@Contended` 注解，由 JVM 自动处理缓存行填充，无需手动添加无用字段。

```java
@sun.misc.Contended
public class PaddedCounter {
    public volatile long count = 0L;
}
```

**原理**：JVM 在编译时会在 `@Contended` 标记的字段前后自动插入填充字节，确保字段独占缓存行。

**启用条件**：
```bash
-XX:-RestrictContended  # 取消限制（JDK 8 默认限制，JDK 9+ 开放）
```

#### 使用 @Contended 的注意事项

1. **仅在 JDK 内部使用**：`@Contended` 位于 `sun.misc` 包，不建议业务代码直接使用。Java 9+ 提供了标准化的 `jdk.internal.vm.annotation.Contended`
2. **内存开销**：每个 `@Contended` 字段会额外占用约 128 字节（前后各 64 字节填充），大量使用时需注意内存
3. **对象头影响**：对象头（12 字节，64 位 JVM 压缩指针）也会影响缓存行对齐

#### 💡 面试点睛

**面试题 1：@Contended 注解的原理是什么？**

回答：JVM 在编译时自动在标记字段前后插入填充字节（padding），确保字段独占一个 64 字节缓存行，消除伪共享。需要 `-XX:-RestrictContended` 启用（JDK 8）。

**面试题 2：@Contended 和手动填充有什么区别？**

回答：@Contended 由 JVM 自动处理，代码更简洁，无需手动添加无用字段；手动填充需要开发者自己计算填充大小，且可能被编译器优化掉。两者目的相同，都是消除伪共享。

## 4. Disruptor 框架剖析

### 4.1 核心组件：RingBuffer、Sequencer、Sequence Barrier

#### RingBuffer

Disruptor 的核心数据结构，一个**环形数组**（预分配、无 GC）：

```java
// RingBuffer 创建
RingBuffer<Event> ringBuffer = new RingBuffer<>(
    Event::new,           // 事件工厂
    new SingleProducerSequencer(1024, new BusySpinWaitStrategy()),
    new BlockingWaitStrategy()
);
```

特点：
- **预分配**：启动时一次性分配所有对象，运行时不创建新对象
- **无 GC**：避免垃圾回收带来的停顿
- **数组实现**：比链表缓存友好（CPU 预取）

#### Sequencer

负责生产者的序列号分配：
- **SingleProducerSequencer**：单生产者，无锁
- **MultiProducerSequencer**：多生产者，使用 CAS

#### Sequence Barrier

消费者协调机制，消费者通过 `SequenceBarrier` 等待可用事件：

```java
SequenceBarrier barrier = ringBuffer.newBarrier();
long availableSequence = barrier.waitFor(nextSequence);
```

#### 💡 面试点睛

**面试题 1：Disruptor 的 RingBuffer 为什么用数组而不是链表？**

回答：数组内存连续，CPU 缓存友好（预取），且可以预分配避免 GC。链表节点分散在内存中，缓存不友好，且运行时创建节点触发 GC。RingBuffer 是固定大小的环形数组，通过序列号模运算定位元素。

---

### 4.2 无锁生产消费模型：claim/commit

#### 单生产者模式

```java
// 生产者
long sequence = ringBuffer.next();  // claim 一个序列号
Event event = ringBuffer.get(sequence);
event.setData(data);
ringBuffer.publish(sequence);       // commit

// 消费者
EventHandler<Event> handler = (event, sequence, endOfBatch) -> {
    // 处理事件
};
```

**关键**：生产者通过 `next()` 获取序列号，消费者通过 `SequenceBarrier` 感知新事件，全程无锁。

#### 多生产者模式

多生产者使用 CAS 竞争序列号，每个生产者写入后设置对应位置的可用标记，消费者检查所有标记后才消费。

#### 💡 面试点睛

**面试题 1：Disruptor 的单生产者和多生产者模式有什么区别？**

回答：单生产者使用 SingleProducerSequencer，序列号分配无锁（直接递增），性能最高；多生产者使用 MultiProducerSequencer，序列号分配用 CAS 竞争，每个生产者写入后设置可用标记，消费者需检查所有标记。单生产者性能显著优于多生产者。

---

### 4.3 为什么这么快：预填充缓存行、无锁、无 GC

| 优化点 | 传统队列 | Disruptor |
|--------|---------|-----------|
| 内存分配 | 运行时 new | 预分配，无 GC |
| 锁机制 | ReentrantLock/阻塞 | 无锁，CAS/内存屏障 |
| 缓存行 | 无处理 | `@Contended` 填充 |
| 数据结构 | 链表（不连续） | 数组（连续，CPU 预取） |
| 批处理 | 单条处理 | 批量消费，减少系统调用 |
| 伪共享 | 存在 | 消除 |

**性能对比**：Disruptor 的吞吐量可达百万 TPS，延迟在微秒级。

#### 💡 面试点睛

**面试题 1：Disruptor 相比 BlockingQueue 性能提升多少？**

回答：Disruptor 吞吐量可达百万 TPS，延迟在微秒级；BlockingQueue（如 LinkedBlockingQueue）吞吐量通常在十万级，延迟在毫秒级。提升来自：无锁设计、预分配无 GC、缓存行填充、数组缓存友好。

## 5. 高并发场景的锁策略选择

### 5.1 synchronized vs ReentrantLock vs StampedLock

| 特性 | synchronized | ReentrantLock | StampedLock |
|------|-------------|---------------|-------------|
| 实现 | JVM 内置（Monitor） | AQS 框架 | AQS 变体 |
| 锁类型 | 独占锁 | 独占/公平/非公平 | 独占/乐观读/悲观读 |
| 可中断 | 否 | 是（lockInterruptibly） | 是 |
| 超时获取 | 否 | 是（tryLock(timeout)） | 是 |
| 条件变量 | 1 个（wait/notify） | 多个（Condition） | 不支持 |
| 性能 | JDK 6+ 优化后接近 ReentrantLock | 优 | 读多写少场景最优 |
| 乐观读 | 不支持 | 不支持 | 支持 |

#### 💡 面试点睛

**面试题 1：synchronized 和 ReentrantLock 的区别？**

回答：synchronized 是 JVM 内置，自动释放锁，不可中断，代码简洁；ReentrantLock 是 API，需手动释放，支持可中断、超时获取、公平锁、多条件变量。JDK 6+ synchronized 优化（偏向锁、轻量级锁、自旋）后性能接近 ReentrantLock。简单场景用 synchronized，复杂场景用 ReentrantLock。

**面试题 2：StampedLock 是什么？有什么优势？**

回答：JDK 8 引入的锁，支持三种模式：写锁、悲观读锁、乐观读。乐观读不加锁，通过版本戳验证数据是否被修改，适合读多写少场景，性能比 ReadWriteLock 更高。但 StampedLock 不支持条件变量，且不支持重入。

---

### 5.2 读多写少极致优化：StampedLock 乐观读

#### StampedLock 三种模式

```java
StampedLock lock = new StampedLock();

// 1. 写锁
long stamp = lock.writeLock();
try {
    // 写操作
} finally {
    lock.unlockWrite(stamp);
}

// 2. 悲观读锁
long stamp = lock.readLock();
try {
    // 读操作
} finally {
    lock.unlockRead(stamp);
}

// 3. 乐观读（无锁读，性能最高）
long stamp = lock.tryOptimisticRead();
// 读取数据
if (!lock.validate(stamp)) {
    // 乐观读失败，升级为悲观读
    stamp = lock.readLock();
    try {
        // 重新读取
    } finally {
        lock.unlockRead(stamp);
    }
}
```

**乐观读原理**：读取时记录一个 stamp（版本戳），读完后验证 stamp 是否变化。如果没变化，说明读期间无写操作，读到的数据有效；如果变化了，说明有写操作，需要升级为悲观读锁重新读取。

**适用场景**：读操作占 90% 以上，且读操作不依赖数据的绝对一致性（可以容忍偶尔的重读）。

#### 💡 面试点睛

**面试题 1：StampedLock 的乐观读怎么实现的？**

回答：读取时记录版本戳 stamp，读完后调用 validate(stamp) 检查 stamp 是否变化。如果没变化，说明读期间无写操作，数据有效；如果变化了，升级为悲观读锁重新读取。全程无锁（乐观读阶段），性能极高。

**面试题 2：StampedLock 有什么缺点？**

回答：不支持条件变量（Condition），不支持重入，乐观读可能失败需要重试。如果读操作需要强一致性，不适合用乐观读。API 较复杂，使用不当容易出错。

### 5.3 ThreadLocalRandom 原理

#### 为什么不用 Random

`java.util.Random` 是线程安全的（内部使用 `AtomicLong` 的 CAS），但高并发下竞争激烈，性能差。

#### ThreadLocalRandom 设计

```java
// 每个线程维护一个独立的种子
public class ThreadLocalRandom extends Random {
    // 线程局部种子，通过 Unsafe 直接操作线程对象字段
    private static final long SEED = Unsafe.getUnsafe().objectFieldOffset(
        Thread.class.getDeclaredField("threadLocalRandomSeed")
    );

    public int nextInt() {
        return mix64(nextSeed());
    }

    private long nextSeed() {
        Thread t = Thread.currentThread();
        long r = UNSAFE.getLong(t, SEED) + GAMMA;
        UNSAFE.putLong(t, SEED, r);
        return r;
    }
}
```

**特点**：
- 每个线程有独立的种子，无竞争
- 通过 `Unsafe` 直接操作 `Thread` 对象的字段，避免 `ThreadLocal` 的哈希查找开销
- 性能是 `Random` 的 10 倍以上

**使用**：
```java
int random = ThreadLocalRandom.current().nextInt(100);
```

#### 💡 面试点睛

**面试题 1：ThreadLocalRandom 为什么比 Random 快？**

回答：Random 使用 AtomicLong 的 CAS，高并发下竞争激烈；ThreadLocalRandom 每个线程有独立种子，通过 Unsafe 直接操作 Thread 字段，无竞争，无哈希查找开销。性能是 Random 的 10 倍以上。

**面试题 2：ThreadLocalRandom 和 ThreadLocal 有什么关系？**

回答：没有关系。ThreadLocalRandom 不使用 ThreadLocal，而是通过 Unsafe 直接操作 Thread 对象的 threadLocalRandomSeed 字段。避免了 ThreadLocal 的哈希查找开销，性能更高。

### 5.4 线程数与 CPU 核心数的关系模型

#### 线程数设置原则

| 场景 | 推荐线程数 | 原因 |
|------|-----------|------|
| CPU 密集型 | NCPU + 1 | 充分利用 CPU，+1 防止偶发阻塞 |
| IO 密集型 | NCPU * 2 ~ NCPU * (1 + W/C) | W=等待时间，C=计算时间 |
| 混合型 | 根据 profiling 调整 | 结合 CPU 和 IO 占比 |

#### 公式

```
最优线程数 = NCPU * (1 + W/C)
```

- W：任务等待时间（IO 等待、锁等待等）
- C：任务计算时间

**示例**：
- 一个任务计算 10ms，等待（IO）90ms，W/C = 9
- 4 核 CPU：4 * (1 + 9) = 40 线程

#### 动态线程池

使用 `ThreadPoolExecutor` 配合监控动态调整：

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    corePoolSize,      // 核心线程数
    maximumPoolSize,   // 最大线程数
    keepAliveTime,     // 空闲线程存活时间
    TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(queueCapacity),
    new ThreadFactoryBuilder().setNameFormat("biz-pool-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
);

// 动态调整
executor.setCorePoolSize(newCoreSize);
executor.setMaximumPoolSize(newMaxSize);
```

#### 💡 面试点睛

**面试题 1：线程池大小怎么设置？**

回答：CPU 密集型：NCPU + 1；IO 密集型：NCPU * (1 + W/C)，W 是等待时间，C 是计算时间。例如任务计算 10ms，IO 等待 90ms，4 核 CPU 最优线程数约 40。实际需根据 profiling 调整。

**面试题 2：线程池的拒绝策略有哪些？**

回答：AbortPolicy（默认，抛异常）、CallerRunsPolicy（由调用线程执行）、DiscardPolicy（静默丢弃）、DiscardOldestPolicy（丢弃队列最老任务）。CallerRunsPolicy 有降级效果，适合防止过载。

- **配合 Arthas**：`thread` 命令查看线程池状态，`trace` 查看任务执行耗时
- **推荐资源**：《Java 并发编程实战》第 8 章（线程池）

## 6. 内存模型与指令重排验证

### 6.1 volatile 保证的底层细节（StoreLoad 屏障）

#### volatile 的语义

1. **可见性**：一个线程修改 volatile 变量，其他线程立即可见
2. **禁止指令重排**：编译器和 CPU 不会对 volatile 变量的读写进行重排序

#### 内存屏障（Memory Barrier/Fence）

JVM 在 volatile 读写前后插入内存屏障：

```
写 volatile：
    [StoreStore 屏障]  // 禁止前面的普通写和 volatile 写重排
    volatile 写
    [StoreLoad 屏障]   // 禁止 volatile 写和后面的 volatile 读/写重排

读 volatile：
    [LoadLoad 屏障]    // 禁止 volatile 读和后面的普通读重排
    volatile 读
    [LoadStore 屏障]   // 禁止 volatile 读和后面的普通写重排
```

**StoreLoad 屏障**：最重的屏障，会使该屏障之前的所有内存访问（读/写）先于屏障之后的内存访问完成。在 x86 上，`StoreLoad` 对应 `mfence` 指令或 `lock` 前缀。

#### volatile 的实现

在 HotSpot VM 中：
- **写 volatile**：生成 `lock addl $0x0, (%rsp)` 指令（空操作，但带 LOCK 前缀，触发缓存一致性协议）
- **读 volatile**：普通读（x86 的内存模型本身较强，读 volatile 无需特殊指令）

#### 💡 面试点睛

**面试题 1：volatile 如何保证可见性？**

回答：写 volatile 时插入 StoreLoad 屏障，触发缓存一致性协议（MESI），使其他核心的缓存失效，强制从主内存读取。x86 上通过 LOCK 前缀实现（如 lock addl $0x0, (%rsp)）。

**面试题 2：volatile 能代替 synchronized 吗？**

回答：不能。volatile 保证可见性和禁止重排，但不保证原子性。例如 volatile++ 不是原子操作（读取-修改-写入三步），多线程下仍会丢失更新。需要原子性时必须用 synchronized 或 Atomic 类。

### 6.2 利用 jcstress 测试并发代码

#### jcstress 简介

jcstress（Java Concurrency Stress Tests）是 OpenJDK 官方提供的并发测试框架，用于验证并发代码的正确性。

#### 使用示例

```java
import org.openjdk.jcstress.annotations.*;
import org.openjdk.jcstress.infra.results.I_Result;

@JCStressTest
@Outcome(id = "1", expect = Expect.ACCEPTABLE, desc = "正常情况")
@Outcome(id = "0", expect = Expect.ACCEPTABLE_INTERESTING, desc = "重排导致")
@State
public class VolatileTest {
    int x;
    volatile int y;

    @Actor
    public void actor1() {
        x = 1;
        y = 1;  // volatile 写，前面的写不能重排到后面
    }

    @Actor
    public void actor2(I_Result r) {
        if (y == 1) {
            r.r1 = x;  // 如果 y==1，x 应该已经是 1
        } else {
            r.r1 = -1;
        }
    }
}
```

**运行**：
```bash
mvn clean install
java -jar target/jcstress.jar -t VolatileTest
```

**结果分析**：
- 如果 `y` 不是 volatile，`x = 1` 可能重排到 `y = 1` 之后，导致 `actor2` 看到 `y==1` 但 `x==0`
- 如果 `y` 是 volatile，`StoreStore` 屏障保证 `x = 1` 先于 `y = 1`，不会出现 `x==0` 的情况

#### 💡 面试点睛

**面试题 1：jcstress 是什么？有什么用？**

回答：jcstress 是 OpenJDK 官方的并发压力测试框架，用于验证并发代码的正确性。通过 @Actor 注解定义多线程操作，检查是否存在指令重排或数据竞争导致的问题。适合验证 volatile、synchronized、CAS 等并发机制的正确性。

- **入口**：jcstress 官方示例（openjdk/jcstress GitHub）
- **配合 JMH**：使用 JMH 做性能测试，jcstress 做正确性测试
- **推荐资源**：JEP 171（JDK 内存模型）、JEP 193（VarHandle）

### 6.3 DCL 单例的完整剖析与正确写法

#### 双重检查锁定（Double-Checked Locking）

**错误写法（JDK 5 之前）**：
```java
public class Singleton {
    private static Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {           // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {   // 第二次检查
                    instance = new Singleton();  // 问题在这里！
                }
            }
        }
        return instance;
    }
}
```

**问题**：`instance = new Singleton()` 不是原子操作，可能重排为：
1. 分配内存
2. 将引用指向内存地址（此时 `instance != null`）
3. 初始化对象

如果步骤 2 和 3 重排，其他线程可能拿到未初始化的对象。

#### 正确写法

**方案 1：volatile 禁止重排（JDK 5+）**：
```java
public class Singleton {
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

**方案 2：静态内部类（推荐，无锁且线程安全）**：
```java
public class Singleton {
    private Singleton() {}

    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

**原理**：JVM 保证类的初始化是线程安全的（`<clinit>` 方法加锁），且延迟加载（首次访问 `Holder` 时才初始化）。

**方案 3：枚举（最简洁，防反射/序列化攻击）**：
```java
public enum Singleton {
    INSTANCE;

    public void doSomething() {
        // 业务逻辑
    }
}
```

#### 💡 面试点睛

**面试题 1：DCL 单例为什么要用 volatile？**

回答：`instance = new Singleton()` 不是原子操作，可能重排为：分配内存 → 引用赋值 → 初始化对象。如果引用赋值后初始化前被其他线程访问，会拿到未初始化的对象。volatile 的 StoreLoad 屏障禁止这种重排，保证初始化完成后才赋值引用。

**面试题 2：静态内部类单例为什么线程安全？**

回答：JVM 保证类的初始化（<clinit>）是线程安全的，多个线程同时访问 Holder 类时，只有一个线程执行初始化，其他线程等待。且是延迟加载（首次访问时才初始化），无锁实现，性能最优。

**面试题 3：枚举单例有什么优势？**

回答：枚举单例由 JVM 保证唯一性，天然防反射攻击（反射无法创建枚举实例）和序列化攻击（反序列化时返回同一实例）。代码最简洁，是实现单例的最佳方式（《Effective Java》推荐）。

#### 源码阅读实战建议

- **入口断点**：打在 `java.lang.Class` 的 `newInstance()` 方法，观察对象实例化过程
- **配合 hsdis**：查看 DCL 单例的汇编代码，确认 volatile 的内存屏障插入
- **jcstress 测试**：编写测试验证 DCL 单例的正确性
- **推荐书籍**：《Effective Java》第 3 条（单例）、《Java 并发编程实战》第 16 章

## 附录：高并发优化速查表

### 原子类选择

| 场景 | 推荐类 | 说明 |
|------|--------|------|
| 低并发计数 | AtomicLong | 简单，支持 CAS 操作 |
| 高并发计数 | LongAdder | 分段累加，吞吐高 |
| 高并发累加（自定义逻辑） | LongAccumulator | 支持自定义二元操作 |
| 并发引用更新 | AtomicReference | CAS 更新对象引用 |
| 带版本号的引用 | AtomicStampedReference | 解决 ABA |
| 布尔标记 | AtomicBoolean | CAS 更新布尔值 |

### 锁选择

| 场景 | 推荐方案 | 说明 |
|------|---------|------|
| 简单同步 | synchronized | JDK 6+ 优化后性能优秀 |
| 需要超时/中断 | ReentrantLock | 功能丰富 |
| 读多写少 | StampedLock 乐观读 | 性能最优 |
| 批量获取/释放 | ReentrantLock + Condition | 多个条件变量 |
| 无锁队列 | ConcurrentLinkedQueue | CAS 实现 |
| 无锁栈 | ConcurrentLinkedDeque | CAS 实现 |

### 并发集合

| 场景 | 推荐集合 | 说明 |
|------|---------|------|
| 高并发 Map | ConcurrentHashMap | 分段锁，读基本无锁 |
| 高并发 List | CopyOnWriteArrayList | 读多写少，写时复制 |
| 高并发 Set | ConcurrentHashMap.newKeySet() | 基于 CHM |
| 阻塞队列 | LinkedBlockingQueue | 生产者-消费者 |
| 延迟队列 | DelayQueue | 定时任务 |
| 优先队列 | PriorityBlockingQueue | 带优先级的阻塞队列 |

### 常用 JVM 参数

```bash
# 线程栈大小
-Xss1m

# 禁用偏向锁（高并发下偏向锁撤销成本高）
-XX:-UseBiasedLocking

# 启用线程本地分配（TLAB）
-XX:+UseTLAB
-XX:TLABSize=256k

# 大页内存（减少 TLB miss）
-XX:+UseLargePages
-XX:LargePageSizeInBytes=2m

# 禁用显式 GC（防止 System.gc()）
-XX:+DisableExplicitGC
```

> **结语**：高并发优化的核心是**减少竞争、消除伪共享、避免阻塞**。从 AtomicLong 到 LongAdder，从 ReentrantLock 到 StampedLock，从传统队列到 Disruptor，每一次演进都是为了在更高并发下保持低延迟和高吞吐。理解底层原理（CPU 缓存、内存屏障、CAS 指令）是做出正确技术选型的基础。
