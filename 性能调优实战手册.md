## 1. CPU 飙高排查实战

### 1.1 问题表现与常见原因

线上 Java 进程突然 CPU 使用率飙升至 100%，且持续不降。此时服务可能仍在运行，但响应极慢，甚至无法处理新请求。这种故障通常由以下原因引起：

- **死循环或大量无休计算**（如 `while(true)` 没有 `sleep`）
- **频繁 Full GC**，GC 线程占用大量 CPU
- **正则表达式回溯**（catastrophic backtracking）
- **多线程在热点方法上激烈竞争**（比如频繁触发的哈希碰撞、锁膨胀）

我们的目标是用最快捷的工具链找出是哪一段代码、哪一个线程在消耗 CPU。

| 原因 | 典型场景 | 排查特征 |
|------|---------|---------|
| **死循环/无终止计算** | `while(true)` 无 `sleep`、递归无终止条件 | 单个线程 CPU 接近 100%，堆栈停在 `RUNNABLE` 的循环方法 |
| **频繁 Full GC** | 内存泄漏导致对象无法回收、堆设置过小 | GC 线程 CPU 高，`jstat` 显示 GC 次数激增 |
| **正则回溯（ReDoS）** | 用户输入触发 `(a+)+` 等嵌套量词 | 线程卡在 `Pattern.match`，CPU 高但无死循环代码 |
| **哈希碰撞/死链** | 自定义 `hashCode()` 返回常量、JDK 1.7 并发扩容 | 多线程卡在 `HashMap.get()` 或 `transfer()`，形成死循环 |
| **JIT 编译器 C2 忙** | 代码刚启动或方法过大触发激进优化 | `C2 CompilerThread` CPU 高，属正常现象 |
| **线程激烈竞争** | 锁粒度过大、大量线程阻塞在 `synchronized` | 多线程状态为 `BLOCKED`，锁膨胀严重 |

#### 💡 面试点睛

**面试题 1：线上 CPU 100%，可能有哪些原因？**

回答要点：死循环/无终止计算（线程 RUNNABLE 停在循环代码）、频繁 Full GC（GC 线程 CPU 高）、正则回溯（线程卡在 Pattern 内部方法）、哈希碰撞（多线程卡在 HashMap 方法）、JIT 编译器忙碌（C2 CompilerThread，通常是短暂的）。

**面试题 2：如何快速区分是业务代码问题还是 GC 问题导致的 CPU 高？**

回答：先用 `jstat -gcutil <PID> 1000` 观察 FGC 列，如果 Full GC 次数在短时间内激增，说明是 GC 问题，应转向内存分析；如果 GC 正常，再用 `top -Hp` 定位高 CPU 线程，通过 `jstack` 查看堆栈定位业务代码。

### 1.2 定位三步法：top -Hp → jstack → 十六进制线程 ID

#### Step 1：找到 CPU 最高的 Java 进程

```bash
top -c
```

找到 Java 进程的 PID，假设为 12345。

#### Step 2：显示该进程的所有线程及 CPU 占用

```bash
top -Hp 12345
```

输出示例：
```
  PID USER     %CPU   COMMAND
12346 app      85.2   java
12347 app       2.0   java
```

此时 12346 是最耗费 CPU 的线程。

#### Step 3：线程 ID 转 16 进制

```bash
printf "%x
" 12346
```

得到 `303a`。

#### Step 4：导出线程栈，搜索该线程

```bash
jstack 12345 | grep -A 20 '0x303a'
```

输出会显示线程名、状态、调用栈，例如：
```
"pool-1-thread-1" #21 prio=5 os_prio=0 tid=0x00007f9a1c001000 nid=0x303a runnable [0x00007f9a0c5f0000]
   java.lang.Thread.State: RUNNABLE
        at com.example.MyService.infiniteLoop(MyService.java:15)
        ...
```

至此，我们直接定位到 `MyService.java` 第 15 行。

#### 补充：用 `jstat` 快速排除 GC 影响

```bash
jstat -gcutil <PID> 1000 5
```

如果 `FGC`（Full GC 次数）在短时间内激增，说明问题在 GC 而非业务代码，应转向内存分析。

#### 💡 面试点睛

**面试题 1：线上 CPU 100%，如何定位？**

回答线索：`top` 定位进程 → `top -Hp` 定位线程 → 转 16 进制 → `jstack` 查看堆栈 → 结合代码分析。同时展示 Arthas 的 `thread -n` 熟练使用，并提及常见原因（死循环、正则回溯、大量 GC）。

**面试题 2：jstack 中的 nid 和 top -Hp 中的 PID 是什么关系？**

回答：nid（Native ID）是线程在操作系统层面的线程 ID，与 `top -Hp` 看到的线程 PID 对应。`printf "%x
" <PID>` 将十进制 PID 转为十六进制，即可在 jstack 输出中搜索 `nid=0x...` 定位线程。

### 1.3 Arthas 一键定位：thread -n 3

Arthas 的 `thread` 命令可以一步到位，无需手动查 PID、做进制转换。

```bash
# 启动 arthas，选择 Java 进程
java -jar arthas-boot.jar

# 查看 CPU 占用最高的前 3 个线程
[arthas@12345]$ thread -n 3
```

输出示例：
```
"pool-1-thread-1" Id=21 cpuUsage=84% deltaTime=204ms time=1835502ms RUNNABLE
    at com.example.MyService.infiniteLoop(MyService.java:15)
    at com.example.MyService.run(MyService.java:10)
    ...

"GC task thread#0" [Internal] cpuUsage=5% deltaTime=3ms time=823ms
    ...

"main" Id=1 cpuUsage=2% WAITING
```

直接看到线程名、代码位置、CPU 占比，并且可以实时刷新（`thread -n 3 -i 2000` 每 2 秒刷新）。

**原理**：`thread -n` 底层调用 `ThreadMXBean#getThreadCpuTime`，在指定间隔（默认 200ms）内获取两次 CPU 时间差，计算增量 CPU 占比。

**专家建议**：`getThreadCpuTime` 属于重型操作。在高负载下，建议增加采样间隔以降低诊断工具对系统的二次伤害：

```bash
thread -n 3 -i 5000    # 每 5 秒刷新一次
```

**进阶技巧**：如果只是想看是否存在死循环，可以用 `thread -b` 查找当前被阻塞的线程，或者用 `monitor -c 5` 统计某个方法调用次数。

#### 💡 面试点睛

**面试题 1：Arthas 的 thread -n 原理是什么？**

回答：底层调用 `ThreadMXBean#getThreadCpuTime`，在指定时间间隔（默认 200ms）内获取两次 CPU 时间差，计算增量 CPU 占比。高负载下建议增加采样间隔（`-i 5000`）降低工具对系统的二次伤害。

**面试题 2：Arthas 和 JDK 自带工具（jstack/jmap）相比有什么优势？**

回答：Arthas 无需手动做 PID 转换、进制转换，一键定位高 CPU 线程；支持实时刷新、方法级监控（watch/trace）、内存分析（heapdump），且无需重启应用。适合生产环境快速诊断。

### 1.4 案例：死循环、频繁 GC、正则回溯、哈希碰撞

#### 案例 1：死循环

**场景**：定时任务中 `while(true)` 忘记加 `sleep`，或递归缺少终止条件。

**Arthas 定位**：
```bash
thread -n 3
# 输出显示某线程 cpuUsage=99% RUNNABLE，停在 while 循环行
```

**修复**：添加终止条件或 `Thread.sleep()`；对于定时任务，改用 `ScheduledExecutorService`。

#### 案例 2：频繁 GC

**场景**：内存泄漏导致对象无法回收，或堆内存设置过小。

**排查**：
```bash
jstat -gcutil <PID> 1000
# 观察 FGC 列是否持续增长
```

**Arthas 辅助**：
```bash
dashboard
# 关注 Memory 区域的 GC 次数和内存使用率
```

若确认 GC 问题，转向第 2 章"内存泄漏排查"或第 4 章"GC 长暂停诊断"。

#### 案例 3：正则回溯（Catastrophic Backtracking / ReDoS）

**场景**：用户输入恶意构造的字符串，触发嵌套量词的正则表达式。

**问题正则示例**：
```java
// 危险！嵌套量词 (a+)+ 在匹配失败时会指数级回溯
Pattern.compile("(a+)+").matcher("aaaaaaaaaaaaaaaaaaaaaaaa!").matches();
```

在 Java 8 环境下，此程序永远不会执行结束，CPU 持续跑满。

**Arthas 定位**：
```bash
thread -n 5
# 输出显示线程卡在 java.util.regex.Pattern$... 内部方法
```

**堆栈特征**：
```
at java.util.regex.Pattern$GroupHead.match(Pattern.java:xxxx)
at java.util.regex.Pattern$Loop.match(Pattern.java:xxxx)
at java.util.regex.Pattern$GroupTail.match(Pattern.java:xxxx)
...
```

**解决方案**：
1. **重写正则**：避免嵌套量词，使用占有型量词（`++`、`*+`）或原子组 `(?>...)`：
   ```java
   // 安全写法：使用占有型量词禁止回溯
   Pattern.compile("(a++)+").matcher(...).matches();
   ```
2. **限制输入长度**：对正则匹配的字符串做长度校验
3. **设置超时**：使用 `RE2/J` 等非回溯正则引擎，或给匹配操作加超时控制
4. **工具检测**：使用 regex101.com 的 debugger 检查回溯步数

#### 案例 4：哈希碰撞与死链

**场景 1：自定义 hashCode 返回常量**

```java
class BadKey {
    @Override
    public int hashCode() { return 1; }  // 所有键映射到同一个桶！
}
```

**后果**：HashMap 退化为链表，查找时间复杂度从 O(1) 降为 O(n)。当数据量达 10 万级时，单次 `get()` 可能需要遍历 10 万次。

**Arthas 定位**：
```bash
trace java.util.HashMap get '#cost>100' -n 10
# 显示 get 方法耗时异常，内部调用大量 Node.next
```

**修复**：重写高质量的 `hashCode()`，确保哈希值均匀分布；使用 `Objects.hash()` 混合字段特征。

**场景 2：JDK 1.7 并发扩容死链（已修复于 1.8，但历史系统仍需警惕）**

JDK 1.7 中 HashMap 扩容采用头插法，多线程并发 `resize()` 时可能形成环形链表，导致 `get()` 死循环，CPU 飙升。

**Arthas 定位**：
```bash
thread -n 5
# 输出显示多个线程卡在 HashMap.transfer() 或 HashMap.get()，状态 RUNNABLE
```

**修复**：升级到 JDK 1.8+；高并发场景使用 `ConcurrentHashMap`。

**场景 3：红黑树退化**

JDK 1.8 引入红黑树优化，但如果 Key 未实现 `Comparable` 接口，且哈希值全部相同，红黑树也无法优化，退化为链表。

#### 💡 面试点睛

**面试题 1：正则回溯怎么发现和解决？**

回答：发现：线程堆栈卡在 `Pattern$GroupHead.match` 等内部方法。解决：使用占有型量词 `++`、原子组 `(?>...)`、限制输入长度、使用 RE2/J 引擎。预防：用 regex101 等工具检测回溯步数。

**面试题 2：HashMap 在什么情况下会退化为链表？**

回答：（1）自定义 `hashCode()` 返回常量或哈希值分布不均；（2）JDK 1.7 并发扩容形成环形链表；（3）JDK 1.8 中 Key 未实现 Comparable 且哈希值全部相同，红黑树无法优化。修复：重写高质量 hashCode、升级 JDK、使用 ConcurrentHashMap。

**面试题 3：线上发现 CPU 高，jstack 显示线程卡在 HashMap.get()，可能是什么原因？**

回答：可能是哈希碰撞导致的链表退化（O(n) 查找），或 JDK 1.7 的并发扩容死链。用 Arthas `trace` 查看 get 方法耗时，检查 Key 的 hashCode 实现，确认 JDK 版本。

### 1.5 易错陷阱：jstack 时间差、线程名混淆、容器环境 PID 问题

#### 陷阱 1：jstack 抓取时机不一致

`top -Hp` 显示的线程 ID 是瞬时值，而 `jstack` 快照可能有几秒延迟。如果线程运行极短（如短时任务），可能捕捉不到。

**解决**：多次抓取（`for i in {1..5}; do jstack <PID> > stack_$i.txt; done`），或使用 Arthas `watch` 持续观察。

#### 陷阱 2：线程名混淆

默认的 `pool-*-thread-*` 名字无法直观看出业务含义。建议创建线程池时自定义 `ThreadFactory`：

```java
ThreadFactory namedFactory = new ThreadFactoryBuilder()
    .setNameFormat("order-process-pool-%d")
    .build();
```

#### 陷阱 3：容器环境 PID 为 1

Docker 内 Java 进程 PID 经常是 1，但 `top` 看到的是宿主机的 PID，需要映射。

**解决**：
- 在容器内执行 `jstack 1`（因为容器内 PID 1 就是 Java 进程）
- 通过挂载 `/proc` 解决：`docker exec -it <container> sh -c "jstack 1"`

#### 陷阱 4：死循环发生在 Native 代码或 GC 线程

`jstack` 只能看到 Java 线程，GC 线程的 CPU 高需要分析 GC 日志。`thread` 命令也会显示 GC 线程，但堆栈信息有限。

**解决**：结合 `jstat -gcutil` 和 GC 日志（`-Xloggc`）分析。

#### 陷阱 5：JIT 编译器误导

C2 CompilerThread 在代码刚启动或方法过大时会占用 CPU，这是正常的 JIT 编译行为，无需处理。

**区分**：CompilerThread 的 CPU 高通常是短暂的，且堆栈显示在 `CompileBroker::compiler_thread_loop`。

#### 💡 面试点睛

**面试题 1：如果 jstack 打印不出来（进程假死），怎么办？**

回答：增加 `-F` 参数强制打印：`jstack -F <PID>`；使用 `jmap -dump` 分析内存配合 `jhat` 或 MAT；用 `gcore` 生成 core dump 后用 `jstack` 分析。进一步说明进程假死通常与 GC、swap 或 IO 阻塞有关。

**面试题 2：Docker 容器中 Java 进程 PID 是 1，如何排查？**

回答：在容器内直接执行 `jstack 1`（容器内 PID 1 就是 Java 进程）；或通过挂载 `/proc` 映射宿主机 PID。也可以用 Arthas 直接 attach 容器内的 Java 进程。

**面试题 3：线程池默认的线程名有什么隐患？**

回答：默认 `pool-*-thread-*` 无法直观看出业务含义，排查问题时难以定位。应使用 ThreadFactoryBuilder 自定义有意义的线程名，如 `order-process-pool-%d`。

#### 源码阅读实战建议

- **入口断点**：打在 `java.lang.Thread` 的 `run()` 方法，观察线程执行流程
- **配合 Arthas**：`watch java.util.regex.Pattern match '{params,returnObj}' -x 3` 观察正则匹配过程
- **配合 jstack**：多次抓取线程栈，对比线程状态变化
- **火焰图**：使用 async-profiler 生成火焰图，直观展示 CPU 热点

## 2. 内存泄漏排查与 OOM 分析

### 2.1 堆内存快照：jmap -dump、MAT 分析

#### 生成堆内存 Dump

```bash
# 方式 1：手动触发（进程假死时也可用）
jmap -dump:format=b,file=/tmp/heap.hprof <PID>

# 方式 2：在 OOM 时自动生成（推荐配置到 JVM 参数）
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/tmp/heap.hprof

# 方式 3：仅 dump 存活对象（减小文件体积）
jmap -dump:live,format=b,file=/tmp/heap.hprof <PID>
```

#### MAT（Memory Analyzer Tool）分析流程

1. **打开 HPROF 文件**：File → Open Heap Dump
2. **运行 Leak Suspects Report**：自动分析可疑泄漏点
3. **查看 Dominator Tree**：按 Retained Heap 排序，找到占用最大的对象
4. **Path to GC Roots**：查看对象到 GC Root 的引用链，定位谁持有该对象
5. **Histogram**：按类统计实例数量和内存占用

**关键指标**：
- **Shallow Heap**：对象自身大小
- **Retained Heap**：对象自身 + 其引用链上所有对象的大小（即释放该对象可回收的内存）

#### 💡 面试点睛

**面试题 1：如何生成堆内存 Dump？**

回答：手动触发用 `jmap -dump:format=b,file=... <PID>`；推荐配置 JVM 参数 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=...` 在 OOM 时自动生成。`live` 参数只 dump 存活对象，减小文件体积。

**面试题 2：MAT 中 Shallow Heap 和 Retained Heap 的区别？**

回答：Shallow Heap 是对象自身大小；Retained Heap 是对象自身加上其引用链上所有对象的大小，即释放该对象可回收的总内存。排查泄漏时关注 Retained Heap 大的对象。

### 2.3 常见泄漏源：ThreadLocal、长生命周期集合、未关闭的流、监听器

#### ThreadLocal 泄漏

**场景**：线程池复用线程，ThreadLocal 未 `remove()`，导致旧任务的数据残留。

**排查**：
```bash
# 查看 ThreadLocalMap 中的 Entry 数量
[arthas@12345]$ vmtool --action getInstances --className java.lang.ThreadLocal$ThreadLocalMap
```

**修复**：
```java
try {
    threadLocal.set(value);
    // 业务逻辑
} finally {
    threadLocal.remove();  // 必须 remove！
}
```

#### 长生命周期集合

**场景**：静态 Map/List 持续添加元素，无清理机制。

**排查**：MAT 中查找 `java.util.HashMap` 或 `java.util.ArrayList` 实例，检查其 size 是否异常大。

**修复**：使用 Guava Cache、Caffeine 等带过期策略的缓存，或定时清理。

#### 未关闭的流/连接

**场景**：数据库连接、文件流、网络连接未关闭，导致相关对象无法回收。

**排查**：MAT 中查找 `java.io.FileInputStream`、`java.sql.Connection` 等实例数量是否持续增长。

**修复**：使用 try-with-resources：
```java
try (InputStream is = new FileInputStream(file)) {
    // 使用流
} // 自动关闭
```

#### 监听器未移除

**场景**：向事件总线、观察者模式注册监听器，但未在对象销毁时移除。

**修复**：在 `destroy()`、`@PreDestroy` 或 `DisposableBean` 中移除监听器。

#### 💡 面试点睛

**面试题 1：ThreadLocal 为什么会内存泄漏？**

回答：ThreadLocalMap 的 Entry 使用弱引用指向 ThreadLocal，但 Value 是强引用。线程池复用线程时，如果未调用 remove()，旧任务的 Value 会一直存在，导致内存泄漏。修复：在 finally 块中调用 threadLocal.remove()。

**面试题 2：静态集合为什么容易导致内存泄漏？**

回答：静态集合的生命周期与 JVM 相同，如果持续添加元素且无清理机制，集合会无限增长。应使用 Guava Cache、Caffeine 等带过期策略的缓存，或定时清理。

**面试题 3：未关闭的资源（流、连接）为什么会导致内存泄漏？**

回答：未关闭的文件流、数据库连接、网络连接会持有底层资源（文件描述符、Socket、数据库连接），相关对象无法被 GC 回收。应使用 try-with-resources 确保资源关闭。

---

### 2.4 元空间 OOM 与直接内存泄漏（-XX:MaxDirectMemorySize）

#### 元空间（Metaspace）OOM

**现象**：`java.lang.OutOfMemoryError: Metaspace`

**原因**：
- 动态生成大量类（如 CGLIB、反射、Groovy 脚本）
- ClassLoader 未释放，导致其加载的类元数据无法回收

**排查**：
```bash
# 查看类加载情况
jstat -class <PID>

# 查看已加载类数量
[arthas@12345]$ classloader -l
```

**调优**：
```bash
-XX:MaxMetaspaceSize=256m
-XX:MetaspaceSize=128m
```

**修复**：检查是否有动态类加载泄漏，确保自定义 ClassLoader 被正确回收。

#### 直接内存（Direct Memory）泄漏

**现象**：`java.lang.OutOfMemoryError: Direct buffer memory`

**原因**：NIO 的 `ByteBuffer.allocateDirect()` 分配的内存不在堆内，由 `Cleaner` 释放。如果引用未释放，直接内存会持续增长。

**排查**：
```bash
# 查看直接内存使用（需开启 JMX）
jcmd <PID> VM.native_memory summary

# 或启动参数加 -XX:NativeMemoryTracking=summary
```

**调优**：
```bash
-XX:MaxDirectMemorySize=512m
```

**修复**：确保 `ByteBuf` 被正确释放（Netty 的 `ReferenceCountUtil.release()`），或使用堆内存缓冲区。

#### 💡 面试点睛

**面试题 1：Metaspace OOM 和 PermGen OOM 有什么区别？**

回答：JDK 8 之前使用 PermGen（永久代），在堆内存中，大小固定；JDK 8+ 使用 Metaspace，在本地内存中，默认无上限（受限于系统内存）。Metaspace OOM 通常是动态生成大量类（CGLIB、反射）或 ClassLoader 未释放导致。

**面试题 2：直接内存泄漏怎么排查？**

回答：现象是 `OutOfMemoryError: Direct buffer memory`。排查：开启 `-XX:NativeMemoryTracking=summary`，用 `jcmd <PID> VM.native_memory summary` 查看直接内存使用。常见于 NIO 的 ByteBuffer.allocateDirect() 未释放，或 Netty 的 ByteBuf 引用计数未归零。

**面试题 3：-XX:MaxDirectMemorySize 不设置会怎样？**

回答：默认与 `-Xmx`（堆最大内存）相同。如果大量使用直接内存（如 Netty、NIO），可能导致直接内存耗尽而堆内存还有很多。建议根据实际使用量显式设置。

### 2.5 陷阱：弱引用泄漏、GC Root 误判

#### 陷阱 1：弱引用泄漏

弱引用（`WeakReference`）在 GC 时会被回收，但如果对象还被强引用链持有，弱引用不会自动清理。

**场景**：`WeakHashMap` 的 Key 被外部强引用持有，导致 Entry 无法回收。

**排查**：MAT 中查看 `WeakHashMap` 的 size 和实际有效 Entry 数量。

#### 陷阱 2：GC Root 误判

MAT 中显示的 GC Root 可能是框架内部的引用（如 Spring 的 BeanFactory），误以为是框架泄漏，实际是业务代码持有。

**解决**：关注 `Path to GC Roots` 中距离最短的路径，通常泄漏点在靠近业务代码的引用层。

#### 陷阱 3：Dump 文件过大

生产环境堆内存可能达几十 GB，dump 文件过大导致分析困难。

**解决**：
- 使用 `jmap -dump:live` 仅 dump 存活对象
- 使用 Arthas `heapdump` 替代，支持压缩
- 使用在线分析工具（如 HeapHero）避免本地加载大文件

#### 💡 面试点睛

**面试题 1：WeakHashMap 会内存泄漏吗？**

回答：会。虽然 Entry 的 Key 是弱引用，但如果 Key 被外部强引用持有，或者 Value 间接引用 Key，会导致 Entry 无法回收。使用时要确保 Key 不被强引用，且 Value 不引用 Key。

**面试题 2：MAT 中 Path to GC Roots 很长，怎么定位泄漏？**

回答：关注距离最短的路径，通常泄漏点在靠近业务代码的引用层。框架内部的引用（如 Spring BeanFactory）通常是正常的，重点看业务对象被谁持有。

#### 源码阅读实战建议

- **入口断点**：打在 `java.lang.ref.Finalizer` 的 `register()` 方法，观察对象 finalize 过程
- **配合 Arthas**：`watch java.lang.ThreadLocal set '{params,returnObj}'` 观察 ThreadLocal 使用
- **配合 MAT**：使用 Path to GC Roots 功能，从泄漏对象反向追踪引用链
- **JOL 工具**：使用 Java Object Layout 查看对象内存布局，理解引用关系

### 3.3 案例：嵌套锁顺序、数据库连接池等待、同步块粒度过大

#### 案例 1：嵌套锁顺序不一致

**场景**：
```java
// 线程 A
synchronized(lockA) {
    synchronized(lockB) { /* ... */ }
}

// 线程 B
synchronized(lockB) {  // 顺序相反！
    synchronized(lockA) { /* ... */ }
}
```

**修复**：全局统一锁的获取顺序，或使用 `tryLock` 避免无限等待。

#### 案例 2：数据库连接池等待

**场景**：事务中持有连接不释放，连接池耗尽，新请求等待连接形成死锁。

**排查**：
```bash
# 查看线程状态
[arthas@12345]$ thread | grep WAITING

# 查看数据库连接池状态
[arthas@12345]$ vmtool --action getInstances --className com.alibaba.druid.pool.DruidDataSource
```

**修复**：
- 缩小事务范围，避免在事务中调用外部 HTTP 接口
- 设置连接超时：`spring.datasource.hikari.connection-timeout=30000`
- 使用 `try-with-resources` 确保连接释放

#### 案例 3：同步块粒度过大

**场景**：在 `synchronized` 块中执行数据库查询、RPC 调用等耗时操作。

**修复**：
```java
// 错误：锁内包含耗时操作
synchronized(this) {
    result = database.query();  // 阻塞其他线程
}

// 正确：仅对必要操作加锁
Data data = database.query();  // 无锁
synchronized(this) {
    cache.put(data.getId(), data);  // 仅对缓存操作加锁
}
```

#### 💡 面试点睛

**面试题 1：如何避免死锁？**

回答：（1）统一锁的获取顺序；（2）使用 tryLock 超时获取；（3）减少锁的持有时间；（4）使用无锁数据结构（ConcurrentHashMap 等）；（5）使用 Lock 的 tryLock 替代 synchronized。

**面试题 2：数据库连接池耗尽导致系统假死，怎么排查？**

回答：用 `thread | grep WAITING` 查看大量线程等待连接；用 Arthas `vmtool` 查看连接池活跃连接数。修复：缩小事务范围、设置连接超时、避免事务中调用外部接口。

### 3.4 优化：缩小锁范围、使用 tryLock、读写锁

#### 缩小锁范围

将锁保护的范围最小化，仅保护共享变量的读写操作。

#### 使用 tryLock 避免死锁

```java
if (lock.tryLock(3, TimeUnit.SECONDS)) {
    try {
        // 执行业务逻辑
    } finally {
        lock.unlock();
    }
} else {
    // 获取锁失败，记录日志或降级处理
    log.warn("获取锁超时，任务放弃");
}
```

#### 读写锁（ReadWriteLock）

读多写少的场景使用 `ReentrantReadWriteLock`：

```java
private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
private final Lock readLock = rwLock.readLock();
private final Lock writeLock = rwLock.writeLock();

// 读操作
readLock.lock();
try {
    return cache.get(key);
} finally {
    readLock.unlock();
}

// 写操作
writeLock.lock();
try {
    cache.put(key, value);
} finally {
    writeLock.unlock();
}
```

#### 无锁化方案

- **CAS**：使用 `AtomicInteger`、`AtomicReference` 等原子类
- **Concurrent 集合**：`ConcurrentHashMap`、`CopyOnWriteArrayList`
- **Disruptor**：基于环形数组的无锁队列

#### 💡 面试点睛

**面试题 1：synchronized 和 ReentrantLock 的区别？**

回答：synchronized 是 JVM 内置，自动释放锁，不可中断；ReentrantLock 是 API，需手动释放，支持可中断、超时获取、公平锁、多条件变量。JDK 6+ synchronized 优化后性能接近 ReentrantLock。

**面试题 2：读写锁适用于什么场景？**

回答：读多写少的场景。读锁之间不互斥，写锁与读锁/写锁互斥。注意：读锁不能升级为写锁（会死锁），写锁可以降级为读锁。

**面试题 3：什么时候应该用无锁方案替代锁？**

回答：高并发、低竞争、简单操作（如计数器）适合 CAS；读多写少适合 CopyOnWriteArrayList；队列适合 ConcurrentLinkedQueue 或 Disruptor。复杂业务逻辑仍需要锁。

#### 源码阅读实战建议

- **入口断点**：打在 `java.util.concurrent.locks.ReentrantLock` 的 `lock()` 方法，观察 AQS 队列构建
- **配合 Arthas**：`thread -b` 查看阻塞线程，`trace java.util.concurrent.locks.Lock lock` 查看锁获取耗时
- **配合 jstack**：分析线程状态分布（RUNNABLE、BLOCKED、WAITING、TIMED_WAITING）
- **推荐书籍**：《Java 并发编程实战》、《Java 并发编程的艺术》

## 4. GC 长暂停问题诊断

### 4.1 GC 日志解读（启用 -Xloggc）

#### 启用 GC 日志（JDK 8）

```bash
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/var/log/gc.log
-XX:+UseGCLogFileRotation
-XX:NumberOfGCLogFiles=10
-XX:GCLogFileSize=100M
```

#### 启用 GC 日志（JDK 9+）

```bash
-Xlog:gc*:file=/var/log/gc.log:time,uptime:filecount=10,filesize=100M
```

#### 日志解读示例

```
[2024-01-15T10:23:45.123+0800] GC(1234) Pause Young (G1 Evacuation Pause) 512M->128M(1024M) 45.234ms
```

- `GC(1234)`：第 1234 次 GC
- `Pause Young`：Young GC（G1 收集器）
- `512M->128M`：GC 前堆内存 512M，GC 后 128M
- `(1024M)`：堆总大小 1024M
- `45.234ms`：GC 暂停时间

#### 💡 面试点睛

**面试题 1：JDK 8 和 JDK 9+ 的 GC 日志参数有什么区别？**

回答：JDK 8 使用 `-XX:+PrintGCDetails -Xloggc:...` 等参数；JDK 9+ 使用统一的 `-Xlog` 语法，如 `-Xlog:gc*:file=...:time,uptime`。JDK 9+ 的日志更灵活，支持标签和级别过滤。

**面试题 2：GC 日志中 512M->128M(1024M) 45.234ms 各字段含义？**

回答：512M 是 GC 前堆内存使用，128M 是 GC 后使用，1024M 是堆总大小，45.234ms 是 GC 暂停时间。如果 GC 后内存下降很少，说明存活对象多，可能需要调整代大小或检查内存泄漏。

---

### 4.2 常见暂停原因：Young GC 晋升失败、Full GC 频繁、大对象分配

#### Young GC 晋升失败（Promotion Failed）

**现象**：Young GC 时，Survivor 区放不下存活对象，直接晋升老年代，但老年代空间不足。

**日志特征**：
```
[GC (Allocation Failure)  512M->480M(1024M), 0.5234567 secs]
[Full GC (Ergonomics)  480M->128M(1024M), 2.3456789 secs]
```

**原因**：
- Survivor 区过小
- 对象晋升阈值（`MaxTenuringThreshold`）设置不当
- 大对象直接分配到老年代

#### Full GC 频繁

**现象**：Full GC 次数多，每次暂停时间长。

**排查**：
```bash
jstat -gcutil <PID> 1000
# 观察 FGC 列和 FGCT（Full GC 总时间）
```

**原因**：
- 内存泄漏
- 堆内存设置过小
- 元空间不足

#### 大对象分配（Humongous Allocation）

**现象**：G1 收集器中，大于 Region 大小 50% 的对象直接分配在 Humongous Region，导致频繁 GC。

**日志特征**：
```
[GC pause (G1 Humongous Allocation) (young) (initial-mark) 512M->256M(1024M) 123.456ms]
```

**修复**：
- 增大 Region 大小：`-XX:G1HeapRegionSize=16m`
- 避免一次性分配大数组/大对象

#### 💡 面试点睛

**面试题 1：Young GC 晋升失败是什么原因？**

回答：Survivor 区过小，放不下存活对象；对象晋升阈值设置不当，对象过早进入老年代；大对象直接分配到老年代导致空间不足。修复：增大 Survivor 区、调整 MaxTenuringThreshold、避免大对象。

**面试题 2：Full GC 频繁怎么排查？**

回答：用 `jstat -gcutil` 观察 FGC 列和 FGCT（Full GC 总时间）。原因可能是内存泄漏、堆设置过小、元空间不足。结合 heapdump 分析内存占用，检查是否有大对象或内存泄漏。

**面试题 3：G1 的 Humongous Allocation 是什么？**

回答：G1 中大于 Region 大小 50% 的对象称为 Humongous 对象，直接分配在 Humongous Region。频繁分配会导致 GC 频繁。修复：增大 G1HeapRegionSize，或拆分大对象。

---

### 4.3 调优案例：晋升阈值、Survivor 区大小、堆分片

#### 案例：晋升阈值调整

```bash
# 默认 15，适当降低可减少老年代压力
-XX:MaxTenuringThreshold=8
```

#### 案例：Survivor 区大小

```bash
# 增大 Survivor 区，避免过早晋升
-XX:SurvivorRatio=8  # Eden:Survivor = 8:1:1
```

#### 案例：堆分片（Heap Fragmentation）

**现象**：老年代有空间，但无连续空间分配大对象，触发 Full GC。

**修复**：
- 使用 G1 收集器（自动处理碎片）
- CMS 收集器开启碎片整理：`-XX:+UseCMSCompactAtFullCollection`

#### 💡 面试点睛

**面试题 1：MaxTenuringThreshold 设置多大合适？**

回答：默认 15，表示对象在 Survivor 区经历 15 次 Young GC 后晋升老年代。如果对象生命周期短，可以设小（如 5-8）减少老年代压力；如果对象生命周期长，保持默认让对象尽快晋升，避免 Survivor 区复制开销。

**面试题 2：SurvivorRatio 怎么调？**

回答：默认 8，表示 Eden:Survivor = 8:1:1（即 Eden 占 80%，两个 Survivor 各占 10%）。如果 Young GC 后存活对象多，可以减小 SurvivorRatio（如 4）增大 Survivor 区；如果存活对象少，可以增大 SurvivorRatio 给 Eden 更多空间。

### 4.4 ZGC/G1 特殊调优参数

#### G1 收集器

```bash
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200          # 目标最大暂停时间
-XX:G1HeapRegionSize=16m          # Region 大小
-XX:InitiatingHeapOccupancyPercent=45  # 触发并发标记的堆占用率
-XX:G1ReservePercent=10          # 保留内存防止晋升失败
```

#### ZGC（JDK 11+）

```bash
-XX:+UseZGC
-XX:+ZGenerational                 # JDK 21+ 启用分代 ZGC
-XX:ZCollectionInterval=5          # 强制 GC 间隔（秒）
```

ZGC 特点：
- 暂停时间 < 10ms，与堆大小无关
- 适合大内存（TB 级）低延迟场景
- 代价：吞吐量略低于 G1，CPU 占用略高

#### 💡 面试点睛

**面试题 1：G1 和 CMS 有什么区别？**

回答：G1 是区域化收集器，将堆划分为 Region，优先回收垃圾最多的 Region，可预测停顿时间；CMS 是老年代并发收集器，标记-清除算法，有内存碎片问题。G1 替代 CMS 成为 JDK 9+ 默认收集器。

**面试题 2：ZGC 的特点和适用场景？**

回答：ZGC 是 JDK 11+ 引入的低延迟收集器，暂停时间 < 10ms 且与堆大小无关。使用染色指针和读屏障实现并发整理。适用大内存（TB 级）、低延迟场景（如金融交易）。代价是吞吐量略低、CPU 占用略高。

**面试题 3：G1 的 MaxGCPauseMillis 一定能达到吗？**

回答：不一定。MaxGCPauseMillis 是目标值，G1 会尽量调整 Region 回收策略来满足，但如果存活对象过多或内存分配过快，仍可能超过。它是一个软目标，不是硬性保证。

- **JVM 参数实验**：在测试环境调整参数，观察 GC 行为变化
- **推荐工具**：async-profiler 生成火焰图，观察 GC 线程 CPU 占用

## 5. IO 瓶颈与网络排查

### 5.1 磁盘 IO 监控：iostat、iotop

#### iostat

```bash
iostat -x 1 10
```

关键指标：
- `%util`：设备利用率，接近 100% 表示饱和
- `await`：平均 IO 等待时间（ms），> 20ms 需关注
- `svctm`：平均服务时间，应接近 `await`

#### iotop

```bash
iotop -o  # 仅显示有 IO 的进程
```

定位哪个进程在大量读写磁盘。

#### 磁盘 IO 优化

- **异步 IO**：使用 `AsynchronousFileChannel` 或 Netty 的异步 IO
- **批量读写**：减少系统调用次数
- **SSD 替换**：机械硬盘随机读性能差，SSD 提升显著
- **RAID 配置**：根据场景选择 RAID 0（性能）或 RAID 10（安全+性能）

#### 💡 面试点睛

**面试题 1：iostat 中 %util 接近 100% 说明什么？**

回答：设备利用率饱和，IO 请求队列堆积。但 %util 高不一定代表性能差（SSD 可以高 util 低延迟），需结合 await（平均等待时间）判断。如果 await > 20ms，说明 IO 成为瓶颈。

**面试题 2：如何定位哪个进程在大量读写磁盘？**

回答：使用 `iotop -o` 查看 IO 占用最高的进程。或者用 `pidstat -d 1` 查看各进程的磁盘 IO 情况。

### 5.2 网络 IO：netstat、ss、TIME_WAIT 堆积

#### netstat / ss

```bash
# 查看连接状态统计
ss -s

# 查看 TIME_WAIT 状态连接
ss -tan state time-wait | wc -l

# 查看 CLOSE_WAIT 状态（可能资源泄漏）
ss -tan state close-wait | wc -l
```

#### TIME_WAIT 堆积

**现象**：大量连接处于 `TIME_WAIT` 状态，端口耗尽，新连接失败。

**原因**：
- 短连接频繁创建/关闭
- 客户端主动关闭连接

**修复**：
```bash
# 内核参数调优
net.ipv4.tcp_tw_reuse = 1        # 复用 TIME_WAIT 连接
net.ipv4.tcp_tw_recycle = 0      # 已废弃，不建议使用
net.ipv4.tcp_fin_timeout = 30    # 缩短 FIN_WAIT_2 超时
net.ipv4.ip_local_port_range = 1024 65535  # 扩大端口范围
```

**代码层面**：使用连接池（HTTP 连接池、数据库连接池），避免短连接。

#### CLOSE_WAIT 堆积

**现象**：大量连接处于 `CLOSE_WAIT`，服务器端未关闭连接。

**原因**：服务端收到 FIN 后未调用 `close()`，通常是代码中未关闭 Socket 或流。

**排查**：
```bash
# 查看 CLOSE_WAIT 对应的进程
ss -tanp state close-wait
```

**修复**：确保 `finally` 块中关闭 Socket/流，或使用 try-with-resources。

#### 💡 面试点睛

**面试题 1：TIME_WAIT 和 CLOSE_WAIT 的区别？**

回答：TIME_WAIT 是主动关闭方等待 2MSL 时间，确保对方收到 ACK；CLOSE_WAIT 是被动关闭方收到 FIN 后等待应用层调用 close()。TIME_WAIT 堆积通常是因为短连接频繁创建关闭；CLOSE_WAIT 堆积通常是应用层未关闭连接。

**面试题 2：大量 TIME_WAIT 怎么解决？**

回答：内核参数调优（tcp_tw_reuse=1、扩大端口范围）；代码层面使用连接池避免短连接。注意 tcp_tw_recycle 已废弃，不建议使用（NAT 环境下会导致问题）。

**面试题 3：大量 CLOSE_WAIT 说明什么问题？**

回答：服务端收到客户端的 FIN 后未调用 close() 关闭连接，通常是代码中未关闭 Socket 或流。排查：用 `ss -tanp state close-wait` 找到对应进程，检查代码中是否缺少 close() 或 try-with-resources。

### 5.3 零拷贝与直接内存的使用验证

#### 零拷贝（Zero-Copy）

**场景**：文件传输、静态资源服务，避免数据在用户态和内核态之间多次拷贝。

**Java 实现**：
```java
// NIO 的 transferTo/transferFrom
FileChannel source = new FileInputStream("source.zip").getChannel();
FileChannel dest = new FileOutputStream("dest.zip").getChannel();
source.transferTo(0, source.size(), dest);  // 零拷贝
```

**验证**：
```bash
# 使用 strace 查看系统调用
strace -f -e sendfile java -jar app.jar

# 应看到 sendfile() 系统调用，而非 read/write
```

#### 直接内存（Direct Buffer）验证

```bash
# 查看直接内存使用
jcmd <PID> VM.native_memory summary

# 或启动参数
-XX:NativeMemoryTracking=summary
-XX:+PrintNMTStatistics
```

**使用场景**：
- NIO 网络传输（Netty 的 ByteBuf 默认使用直接内存）
- 大文件读写
- 避免堆内存和直接内存之间的数据拷贝

**注意**：直接内存不受 GC 管理，需手动释放或依赖 Cleaner 机制。

#### 💡 面试点睛

**面试题 1：零拷贝的原理是什么？Java 怎么实现？**

回答：零拷贝避免数据在用户态和内核态之间多次拷贝。Java 通过 FileChannel.transferTo() 实现，底层调用 sendfile() 系统调用，数据直接从内核缓冲区发送到网卡，不经过用户态。适合大文件传输场景。

**面试题 2：直接内存（Direct Memory）和堆内存的区别？**

回答：直接内存分配在堆外，不受 GC 管理，适合 NIO 和大文件传输。分配和回收成本较高（需系统调用），但避免了堆内存和直接内存之间的数据拷贝。需注意释放，否则会导致 OutOfMemoryError: Direct buffer memory。

**面试题 3：Netty 为什么使用堆外内存？**

回答：Netty 使用堆外内存（Direct Buffer）进行网络 IO，避免数据在堆内存和直接内存之间的拷贝（零拷贝）。配合引用计数精确控制内存释放，减少 GC 停顿。PooledByteBufAllocator 实现内存池化管理，提高分配效率。

#### 源码阅读实战建议

- **入口断点**：打在 `sun.nio.ch.FileChannelImpl` 的 `transferTo()` 方法，观察零拷贝实现
- **配合 strace**：`strace -f -e sendfile java -jar app.jar` 验证零拷贝系统调用
- **配合 jcmd**：`jcmd <PID> VM.native_memory summary` 查看直接内存使用
- **推荐书籍**：《深入理解 Linux 内核》、《Linux 性能优化实战》

## 6. Arthas 深度用法

### 6.1 watch 观察方法入参/返回值/异常

```bash
# 观察方法入参和返回值
[arthas@12345]$ watch com.example.UserService getUser '{params,returnObj}' -x 2

# 观察异常信息
[arthas@12345]$ watch com.example.UserService getUser '{params,throwExp}' -e

# 观察特定条件（当 userId > 100 时）
[arthas@12345]$ watch com.example.UserService getUser '{params,returnObj}' 'params[0]>100'

# 观察入参深度展开（-x 指定展开层级）
[arthas@12345]$ watch com.example.UserService getUser '{params,returnObj}' -x 3
```

**参数说明**：
- `-x`：展开层级，默认 1
- `-b`：方法执行前观察
- `-e`：异常时观察
- `-s`：方法返回后观察
- `-f`：方法结束后观察（包含正常返回和异常）

#### 💡 面试点睛

**面试题 1：Arthas watch 命令的 -b、-s、-e 参数分别代表什么？**

回答：-b（before）方法执行前观察；-s（success）方法返回后观察；-e（exception）异常时观察；-f（finish）方法结束后观察。可以组合使用，如 `-b -s` 观察入参和返回值。

**面试题 2：watch 命令中 '{params,returnObj}' 是什么语法？**

回答：这是 OGNL 表达式。params 表示方法入参数组，returnObj 表示返回值，throwExp 表示异常对象。可以组合观察多个值，如 '{params[0],returnObj,throwExp}'。

### 6.2 trace 追踪调用链路与耗时

```bash
# 追踪方法耗时，显示调用路径
[arthas@12345]$ trace com.example.OrderService createOrder

# 只显示耗时超过 100ms 的调用

# 追踪多层调用（最多 3 层）
[arthas@12345]$ trace com.example.OrderService createOrder -n 3

[arthas@12345]$ trace java.net.URLClassLoader findClass --skipJDKMethod false
```

```
    `---[45.234ms] com.example.OrderService:createOrder()
        +---[5.123ms] com.example.OrderDao:insert() #23
        +---[30.456ms] com.example.PaymentService:pay() #25
        |   `---[28.789ms] com.example.PaymentGateway:charge() #45
        `---[9.655ms] com.example.NotificationService:send() #27
```

#### 💡 面试点睛

**面试题 1：trace 和 watch 有什么区别？**

回答：trace 追踪方法的调用链路和每层的耗时，适合定位性能瓶颈；watch 观察方法的入参、返回值和异常，适合调试业务逻辑。trace 会显示子方法调用，watch 只关注当前方法。

**面试题 2：trace 中 '#cost>100' 是什么意思？**

回答：只显示耗时超过 100ms 的方法调用，过滤掉正常耗时的调用，快速定位慢方法。可以配合 -n 限制层级，避免输出过多。

### 6.3 vmtool 强制调用对象方法

```bash
# 获取 Spring 容器中的 Bean 并调用方法
```

#### 💡 面试点睛

**面试题 1：vmtool 有什么危险操作？**

回答：vmtool 可以强制调用对象的任意方法，包括修改数据、触发副作用操作。在线上环境使用时需谨慎，避免调用写操作或触发业务流程。建议先在测试环境验证。

**面试题 2：vmtool 和 jmap 有什么区别？**

回答：vmtool 是运行时操作，可以获取对象实例并调用方法；jmap 是静态分析，生成堆 dump 文件。vmtool 适合快速验证和调试，jmap 适合深度内存分析。
